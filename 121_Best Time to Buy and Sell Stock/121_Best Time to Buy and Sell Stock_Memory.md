## 記憶體位址與陣列索引詳解

## 1. 陣列記憶體佈局

### 基礎陣列信息
- **陣列名稱**: `prices`
- **元素型別**: `int` (4 bytes per element)
- **陣列大小**: 6 elements
- **基地址**: `0xA000`
- **總佔用記憶體**: 24 bytes (6 × 4)
- **結束地址**: `0xA017`

### 位址表

```
prices[0] = 7    ──→  0xA000
prices[1] = 1    ──→  0xA004
prices[2] = 5    ──→  0xA008
prices[3] = 3    ──→  0xA00C
prices[4] = 6    ──→  0xA010
prices[5] = 4    ──→  0xA014
```

## 3. 陣列索引版對應

### 記憶體位址公式

```
address = base_address + (index × sizeof(int))
address = 0xA000 + (index × 4)
```

### 索引到位址的映射表

```
┌────────────┬───────────────────┬──────────────────┐
│   Index    │    計算式          │    位址          │
├────────────┼───────────────────┼──────────────────┤
│   [0]      │ 0xA000 + 0×4      │    0xA000        │
│   [1]      │ 0xA000 + 1×4      │    0xA004        │
│   [2]      │ 0xA000 + 2×4      │    0xA008        │
│   [3]      │ 0xA000 + 3×4      │    0xA00C        │
│   [4]      │ 0xA000 + 4×4      │    0xA010        │
│   [5]      │ 0xA000 + 5×4      │    0xA014        │
└────────────┴───────────────────┴──────────────────┘
```

### 陣列存取時間複雜度

- **時間複雜度**: O(1) - 常數時間
- **存取方式**: 直接計算位址，不需迴圈搜尋
- **CPU 指令**: 1-2 個指令完成位址計算 (ADD + LOAD)

## 4. 搜尋／操作過程追蹤

#### 演算法邏輯
```
max_profit = 0
min_price = prices[0] = 7

for i = 1 to 5:
    profit = prices[i] - min_price
    max_profit = max(max_profit, profit)
    min_price = min(min_price, prices[i])
```

#### 快取效能分析

```
優點 (Excellent Cache Behavior):
├─ Stride-1 存取: 每個指令訪問相鄰的記憶體位置
├─ Cache Prefetcher 友善: CPU 可自動預測下一個訪問地址
│  （現代 CPU 有 stride prefetcher，可檢測固定 stride 模式）
├─ Cache Line 利用效率高: 一行 cache (通常 64B) 包含 16 個 int
│  約只需 1 次快取不命中可取得 16 個連續元素
├─ 單一 pass: 每個快取行只被觸及一次，無重複訪問
├─ 分支預測友善: 簡單迴圈，分支預測器可完美預測
└─ 指令級並行高: 連續訪問可互相依存，不會創建長依存鏈

記憶體頻寬利用率: ≈ 95%
快取命中率 (L1 cache): ≈ 100% (假設陣列適應 L1)
平均訪問延遲: ≈ 1 cycle (L1 命中) + prefetch overhead
```

#### 快取效能分析

```
缺點 (Poor Cache Behavior):
├─ Pointer Chasing: 必須訪問一個節點才能獲知下一個節點地址
│  ├─ 無法預測下一個訪問地址 ❌
│  ├─ CPU Prefetcher 無能為力（無確定 stride）
│  └─ 每個節點訪問都是 dependent load（必須等待前一個完成）
├─ 記憶體存取延遲串聯:
│  ├─ Load latency: ≈ 100-200 cycles (L3 miss to main memory)
│  ├─ 6 個節點 = 6 × 100 cycles = 600+ cycles 等待時間
│  └─ 相比陣列的 ≈ 6 cycles
├─ 快取行浪費:
│  ├─ 每次訪問只用 4 bytes (price)，丟棄 60 bytes
│  ├─ Cache footprint 增大，L1/L2 miss 頻率增加
│  └─ 可能污染 TLB (translation lookaside buffer)
├─ 分支預測失敗:
│  ├─ next 指標值無法預測
│  └─ 迴圈終止條件難以預測
└─ 隱藏記憶體延遲困難:
   └─ 無法並行訪問多個節點（dependent chain）

記憶體頻寬利用率: ≈ 5-10%
快取命中率 (L1 cache): ≈ 10-20%
平均訪問延遲: ≈ 100+ cycles (L3 miss)
整體性能: ≈ 50-100倍 slower than array
```

### B. 遞迴實現 (Hypothetical Divide-and-Conquer)

#### 遞迴分治演算法概念

```c
int maxProfit_Recursive(int* prices, int start, int end,
                       int min_price, int max_profit) {
    // Base case
    if (start > end) {
        return max_profit;
    }

    // Recursive case
    int profit = prices[start] - min_price;
    max_profit = max(max_profit, profit);
    min_price = min(min_price, prices[start]);

    // Recur to next element
    return maxProfit_Recursive(prices, start + 1, end,
                              min_price, max_profit);
}
```

## 總結

### 關鍵記憶體概念

| 概念 | 說明 | 本問題的應用 |
|------|------|------------|
| **基地址計算** | `address = base + index × size` | `0xA000 + i × 4` |
| **快取局部性** | 順序存取相鄰記憶體 | Stride-1 訪問，高效率 |
| **堆疊 vs 堆積** | 局部變數/參數在堆疊，動態陣列在堆積 | prices 在堆積，min_price 等在堆疊 |
| **指標追蹤** | 跟蹤值的變化歷史 | min_price、max_profit 的更新過程 |
| **演算法選擇** | 迭代 vs 遞迴的權衡 | 迭代版本優勢明顯 |

### 學習要點

1. **陣列的線性記憶體特性** 使其天然適合快取優化
2. **Sequential access pattern** 允許 CPU prefetcher 預測下一步
3. **O(1) 空間複雜度** 表示堆疊使用最少
4. **一次 pass 遍歷** 使每個快取行只被觸及一次
5. **迭代實現** 遠優於不必要的遞迴
