# LeetCode 49 - Group Anagrams 記憶體分析

## 概述
本文件詳細分析 LeetCode 49 Group Anagrams 的 C 語言解法中的記憶體佈局、資料結構配置、演算法執行追蹤，及其與雜湊表方案的效能比較。

## 4. 搜尋／操作過程追蹤

### Example 1: ["eat", "tea", "tan", "ate", "nat", "bat"]

#### Phase 1: 字串排序

對 sorted_strs[] 中的每個字串進行字元排序：

```
sorted_strs[0]: "eat" ──▶ sort chars ──▶ "aet"
                 ├─ 'e', 'a', 't'
                 └─ 'a', 'e', 't' (sorted)

sorted_strs[1]: "tea" ──▶ sort chars ──▶ "aet"
                 ├─ 't', 'e', 'a'
                 └─ 'a', 'e', 't' (sorted)

sorted_strs[2]: "tan" ──▶ sort chars ──▶ "ant"
                 ├─ 't', 'a', 'n'
                 └─ 'a', 'n', 't' (sorted)

sorted_strs[3]: "ate" ──▶ sort chars ──▶ "aet"
                 ├─ 'a', 't', 'e'
                 └─ 'a', 'e', 't' (sorted)

sorted_strs[4]: "nat" ──▶ sort chars ──▶ "ant"
                 ├─ 'n', 'a', 't'
                 └─ 'a', 'n', 't' (sorted)

sorted_strs[5]: "bat" ──▶ sort chars ──▶ "abt"
                 ├─ 'b', 'a', 't'
                 └─ 'a', 'b', 't' (sorted)
```

#### Phase 2: qsort 排序 pairs[]

```
初始 pairs[] (按 sorted 鍵)：
┌─────┬──────┬──────┐
│ idx │ sort │ orig │
├─────┼──────┼──────┤
│ 0   │ "aet"│ "eat"│
│ 1   │ "aet"│ "tea"│
│ 2   │ "ant"│ "tan"│
│ 3   │ "aet"│ "ate"│
│ 4   │ "ant"│ "nat"│
│ 5   │ "abt"│ "bat"│
└─────┴──────┴──────┘

qsort 比較步驟（逐層分割，歸併排序示意）：
Partition 1: pivot = "aet" (pairs[0])
  ├─ < pivot:  ["abt"] (idx 5)
  ├─ = pivot:  ["aet", "aet", "aet"] (idx 0,1,3)
  └─ > pivot:  ["ant", "ant"] (idx 2,4)

排序後結果：
┌─────┬──────┬──────┐
│ idx │ sort │ orig │
├─────┼──────┼──────┤
│ 0   │ "abt"│ "bat"│  ◄── 新位置 0 (從原索引 5)
│ 1   │ "aet"│ "eat"│  ◄── 新位置 1 (從原索引 0)
│ 2   │ "aet"│ "tea"│  ◄── 新位置 2 (從原索引 1)
│ 3   │ "aet"│ "ate"│  ◄── 新位置 3 (從原索引 3)
│ 4   │ "ant"│ "tan"│  ◄── 新位置 4 (從原索引 2)
│ 5   │ "ant"│ "nat"│  ◄── 新位置 5 (從原索引 4)
└─────┴──────┴──────┘
```

#### Phase 3: 線性掃描分組

```
current_sorted = NULL
current_group = []
result_idx = 0

步驟 1: pairs[0]
  ├─ pairs[0].sorted = "abt"
  ├─ current_sorted = NULL
  ├─ "abt" != NULL ──▶ 新群組開始
  ├─ current_sorted = "abt"
  ├─ current_group = ["bat"]
  └─ result[0] = ["bat"]

步驟 2: pairs[1]
  ├─ pairs[1].sorted = "aet"
  ├─ current_sorted = "abt"
  ├─ strcmp("aet", "abt") = 1 (不相等) ──▶ 新群組
  ├─ result_idx++ ──▶ result_idx = 1
  ├─ current_sorted = "aet"
  ├─ current_group = ["eat"]
  └─ result[1] = ["eat"]

步驟 3: pairs[2]
  ├─ pairs[2].sorted = "aet"
  ├─ current_sorted = "aet"
  ├─ strcmp("aet", "aet") = 0 ──▶ 追加到群組
  ├─ current_group.append("tea")
  └─ result[1] = ["eat", "tea"]

步驟 4: pairs[3]
  ├─ pairs[3].sorted = "aet"
  ├─ current_sorted = "aet"
  ├─ strcmp("aet", "aet") = 0 ──▶ 繼續追加
  ├─ current_group.append("ate")
  └─ result[1] = ["eat", "tea", "ate"]

步驟 5: pairs[4]
  ├─ pairs[4].sorted = "ant"
  ├─ current_sorted = "aet"
  ├─ strcmp("ant", "aet") = 1 (不相等) ──▶ 新群組
  ├─ result_idx++ ──▶ result_idx = 2
  ├─ current_sorted = "ant"
  ├─ current_group = ["tan"]
  └─ result[2] = ["tan"]

步驟 6: pairs[5]
  ├─ pairs[5].sorted = "ant"
  ├─ current_sorted = "ant"
  ├─ strcmp("ant", "ant") = 0 ──▶ 追加到群組
  ├─ current_group.append("nat")
  └─ result[2] = ["tan", "nat"]

最終結果：
result[0] = ["bat"]           ◄── 1 個元素
result[1] = ["eat", "tea", "ate"]  ◄── 3 個元素
result[2] = ["tan", "nat"]    ◄── 2 個元素
```

## 5. 指標變數追蹤總表

### 逐步執行表（Example 1）

| 步驟 | pairs[i].sorted | pairs[i].original | strcmp 結果 | current_sorted | 動作 | 所屬群組 |
|------|-----------------|------------------|-----------|----------------|------|---------|
| 0 (init) | - | - | - | NULL | 初始化 | - |
| 1 | "abt" | "bat" | ✓ != NULL | "abt" | **新群組開始** | result[0] |
| 2 | "aet" | "eat" | -1 (aet > abt) | "aet" | **新群組開始** (result_idx→1) | result[1] |
| 3 | "aet" | "tea" | 0 (aet == aet) | "aet" | **追加到群組** | result[1] |
| 4 | "aet" | "ate" | 0 (aet == aet) | "aet" | **追加到群組** | result[1] |
| 5 | "ant" | "tan" | 1 (ant > aet) | "ant" | **新群組開始** (result_idx→2) | result[2] |
| 6 | "ant" | "nat" | 0 (ant == ant) | "ant" | **追加到群組** | result[2] |

### strcmp 返回值解釋

```
strcmp(a, b) 返回值：
  < 0   : a < b  (字典序 a 在 b 之前)
    = 0   : a == b (完全相同)
  > 0   : a > b  (字典序 a 在 b 之後)

本例中：
  strcmp("aet", "abt") = 1    (e > b at position 1)
  strcmp("aet", "aet") = 0    (完全相同)
  strcmp("ant", "aet") = 1    (n > e at position 1)
  strcmp("ant", "ant") = 0    (完全相同)
```

## 6. 記憶體存取模式比較

### 方法 A: qsort 式排序分組（當前方案）

#### 記憶體配置

```
heap 配置：
┌─────────────────────────────────────────────┐
│ strs[]          (6 個指標)      = 48 bytes  │
│ sorted_strs[]   (6 個指標)      = 48 bytes  │
│ 排序字串副本    (6 個字串)      ≈ 18 bytes │
│ pairs[]         (6 個結構)      = 96 bytes │
│ result[][][]    (分組結果)      ≈ 48 bytes │
│ col_sizes[]     (3 個整數)      = 12 bytes │
├─────────────────────────────────────────────┤
│ 總計：≈ 270 bytes                           │
└─────────────────────────────────────────────┘

時間複雜度：
  - 字串排序: O(n * k log k)  [n 個字串，每個 k 個字符]
  - qsort pairs: O(n log n * k)  [n 個 pair，比較 O(k)]
  - 線性掃描: O(n)
  ────────────────────────────
  總計: O(n * k log k + n log n * k)
```

#### 快取行為

```
L1 快取友好性：中等
┌──────────────────────────────────────────┐
│ 優點：                                    │
│ • 線性掃描分組 ──▶ 順序存取 (好)         │
│ • 排序字串位置集中                        │
│                                          │
│ 缺點：                                    │
│ • qsort 比較時需跳轉                     │
│ • 雙重指標間接存取 (sorted, original)    │
│ • pairs[] 與 strs[] 記憶體分散           │
└──────────────────────────────────────────┘

存取模式示意：
qsort 比較階段：
  pairs[left].sorted  ◄── 跳躍存取
      │
      ├─ 比較字串內容 (k 次字符比較)
      │
      └─ 交換 pairs[left] ◄── 非連續寫

pairs[left] ↔ pairs[right]  (兩個 16-byte 結構交換)
  ├─ 寫 L1 快取污染
  └─ 分支預測困難
```

#### 記憶體配置效率

| 項目 | 開銷 | 說明 |
|------|------|------|
| 指標級聯 | 高 | sorted → 字串, original → 字串 (2 級間接) |
| 配置次數 | 中 | malloc 5-6 次 (字串 + pairs + result) |
| 碎片風險 | 中 | 排序字串副本與原始字串分離 |
| 快取行為 | 中 | qsort 隨機存取，線性掃描順序 |

### 方法 B: 雜湊表方案（對比參考）

#### 雜湊表結構

```c
typedef struct {
    char* sorted;      // 排序鍵
    char** group;      // 同義詞組
    int group_size;
} HashBucket;

// 雜湊表
HashBucket hash_table[HASH_SIZE];
```

#### 記憶體配置

```
heap 配置：
┌─────────────────────────────────────────────────┐
│ strs[]              (6 個指標)        = 48 bytes│
│ sorted_strs[]       (6 個指標)        = 48 bytes│
│ 排序字串副本        (6 個字串)        ≈ 18 bytes│
│ hash_table[]        (HASH_SIZE 項)    = 4K-16K │
│ 鏈表節點或開放尋址   (每項額外開銷)   = 可變    │
│ result[][][]        (分組結果)        ≈ 48 bytes│
├─────────────────────────────────────────────────┤
│ 總計：≈ 4.3K-16.3K bytes                        │
│     (取決於衝突解決策略)                        │
└─────────────────────────────────────────────────┘

時間複雜度：
  - 字串排序: O(n * k log k)  [n 個字串，每個 k 個字符]
  - 雜湊表插入: O(n) 平均 / O(n²) 最壞  [每次 O(k) 雜湊]
  ────────────────────────────
  總計: O(n * k log k + n*k) 平均
       O(n * k log k + n²) 最壞
```

#### 快取行為

```
L1 快取友好性：低 (衝突多) → 高 (衝突少)
┌──────────────────────────────────────────┐
│ 優點：                                    │
│ • 無需 qsort (避免隨機存取)              │
│ • 平均 O(1) 查詢 ──▶ 快速分組            │
│ • 預取友好 (預測下一個桶)                │
│                                          │
│ 缺點：                                    │
│ • 衝突時需遍歷鏈表 (快取缺失)            │
│ • 更大的初始記憶體開銷                    │
│ • 雜湊函式計算額外成本                    │
└──────────────────────────────────────────┘

存取模式示意：
直接尋址 (無衝突)：
  hash(sorted_key) ──▶ bucket_index
                        │
                        ├─ 一次 L1 快取命中
                        └─ 直接存取 (O(1))

衝突鏈遍歷 (衝突多)：
  hash(sorted_key) ──▶ bucket_index
                        │
                        ├─ 第 1 次衝突 ──▶ L1 命中
                        ├─ 第 2 次衝突 ──▶ L1 miss (指標跳轉)
                        ├─ 第 3 次衝突 ──▶ L2 存取
                        └─ ...
```

#### 記憶體配置效率

| 項目 | 開銷 | 說明 |

## 7. 結論

### 推薦場景

#### qsort 方案最佳
- **輸入規模**: n ≤ 10,000
- **字串長度**: k ≤ 20
- **記憶體約束**: 嚴格 (嵌入式/物聯網)
- **快取行為**: 線性掃描最終階段友好
- **實現複雜度**: 低 (標準庫 qsort)
- **預測性**: 確定性 (無衝突)

#### 雜湊表方案最佳
- **輸入規模**: n > 10,000
- **字串長度**: k > 20
- **記憶體充裕**: 伺服器/高效能計算
- **衝突控制**: 良好的雜湊函式
- **時間敏感**: 毫秒級要求
- **動態性**: 頻繁新增/刪除

### 實現品質評估

| 維度 | qsort 方案 | 雜湊表方案 |
|------|---------|---------|
| **時間複雜度** | O(nk log k + nk log n) | O(nk log k + nk) 平均 |
| **空間複雜度** | O(nk) | O(n) 雜湊表 |
| **快取效率** | 中-高 | 中-低 (衝突敏感) |
| **實現簡度** | ★★★★★ | ★★★☆☆ |
| **穩定性** | 完全確定 | 取決於衝突 |
| **適用場景** | 競賽/受限環境 | 生產級別應用 |

### 記憶體效率排名

```
1. qsort 方案 (最緊湊)
   ├─ 無額外開銷結構
   ├─ 指標複用現有字串
   └─ 總開銷: ≈ 0.3% 的輸入大小

2. 優化雜湊表 (中等)
   ├─ 開放尋址 (無鏈表)
   ├─ 動態調整因子
   └─ 總開銷: ≈ 3-5% 的輸入大小

3. 鏈表雜湊表 (最散)
   ├─ 每衝突一個節點
   ├─ 指標級聯
   └─ 總開銷: ≈ 10-20% 的輸入大小
```

## 參考資料

### 關鍵演算法步驟
1. **字串規範化**: 對每個輸入字串排序字符
2. **配對構造**: (排序字串, 原始字串) 對
3. **全局排序**: qsort 按排序字串比較
4. **分組識別**: 線性掃描相同排序字串
5. **結果組裝**: 蒐集原始字串到群組

### 時間/空間成本項
- 字符排序: O(k log k) × n
- 比較排序: O(log n) 層次 × O(n log n) 比較 × O(k) 字串比較 = O(nk log² n)
- 線性掃描: O(n) × O(k) 比較 = O(nk)
- 記憶體: O(n) 原始 + O(nk) 排序副本 + O(n) 輸出結構
