# 480 滑動視窗中位數 - 記憶體分析

## 1. Struct 記憶體佈局

### 1.1 Heap 結構體

```
struct Heap {
    int* data;      // 指標 (8 bytes on 64-bit)
    int size;       // 整數 (4 bytes)
    int capacity;   // 整數 (4 bytes)
    bool isMinHeap; // 布林 (1 byte)
};
```

**欄位偏移與大小：**
```
┌─────────────────────────────────────────┐
│ 位置    │ 欄位名稱    │ 大小     │ 備註   │
├─────────────────────────────────────────┤
│ +0      │ data        │ 8 bytes  │ 指標   │
│ +8      │ size        │ 4 bytes  │ 整數   │
│ +12     │ capacity    │ 4 bytes  │ 整數   │
│ +16     │ isMinHeap   │ 1 byte   │ 布林   │
│ +17     │ (padding)   │ 7 bytes  │ 對齊   │
└─────────────────────────────────────────┘

總大小：24 bytes (含對齊 padding)
```

**記憶體圖示（假設 Heap 在 0xA000）：**
```

       │  data (指標)   │  8 bytes
       ├────────────────┤

       ├────────────────┤

       └────────────────┘
0xA018
```

## 3. 陣列索引版對應

### 3.1 二進制堆陣列索引公式

對於陣列索引 `i` 的元素：
- **左子節點索引**：`2*i + 1`
- **右子節點索引**：`2*i + 2`
- **父節點索引**：`(i - 1) / 2`

### 3.2 第一個視窗 [1,3,-1] 的索引關係表

**小堆 (MaxHeap) 索引表：**
```
┌──────┬────────┬──────────┬──────────┬────────────┐
│ 索引 │ 值     │ 父節點   │ 左子節點 │ 右子節點   │
├──────┼────────┼──────────┼──────────┼────────────┤
│ 0    │ 3      │ N/A      │ 1        │ 2          │
├──────┼────────┼──────────┼──────────┼────────────┤
│ 1    │ 1      │ 0        │ 3        │ 4          │
├──────┼────────┼──────────┼──────────┼────────────┤
│ 2    │ (空)   │ 0        │ 5        │ 6          │
├──────┼────────┼──────────┼──────────┼────────────┤
│ 3-4  │ (空)   │ 1        │ -        │ -          │
└──────┴────────┴──────────┴──────────┴────────────┘

堆特性：small.data[0] = 3 >= small.data[1] = 1 ✓
樹形：
         3(idx:0)
        /
       1(idx:1)
```

**大堆 (MinHeap) 索引表：**
```
┌──────┬────────┬──────────┬──────────┬────────────┐
│ 索引 │ 值     │ 父節點   │ 左子節點 │ 右子節點   │
├──────┼────────┼──────────┼──────────┼────────────┤
│ 0    │ -1     │ N/A      │ 1        │ 2          │
├──────┼────────┼──────────┼──────────┼────────────┤
│ 1-2  │ (空)   │ 0        │ -        │ -          │
└──────┴────────┴──────────┴──────────┴────────────┘

堆特性：large.data[0] = -1 <= 所有子節點 ✓
```

### 3.3 位址到索引映射表

```
┌────────┬───────────┬────────────┬────────────┐
│ 堆類型 │ 位址 (0x) │ 陣列索引   │ 值         │
├────────┼───────────┼────────────┼────────────┤
│ small  │ 0x2000    │ [0]        │ 3          │
│        │ 0x2004    │ [1]        │ 1          │
│        │ 0x2008    │ [2]        │ (空)       │
│        │ 0x200C    │ [3]        │ (空)       │
├────────┼───────────┼────────────┼────────────┤
│ large  │ 0x2100    │ [0]        │ -1         │
│        │ 0x2104    │ [1]        │ (空)       │
│        │ 0x2108    │ [2]        │ (空)       │
│        │ 0x210C    │ [3]        │ (空)       │
└────────┴───────────┴────────────┴────────────┘

位址計算：small.data + (sizeof(int) * idx)
例：small.data[1] 位址 = 0x2000 + (4 * 1) = 0x2004
```

## 4. 搜尋／操作過程追蹤

### 範例 1：`nums = [1,3,-1,-3,5,3,6,7], k = 3`

追蹤所有 6 個視窗位置的完整操作。

#### **步驟 1：視窗 [1, 3, -1] (索引 0-2)**

**初始化：**
```
視窗：[1, 3, -1]
操作序列：insert(1), insert(3), insert(-1)
```

**Insert 1：**
```
操作：
├─ 將 1 添加到 small (MaxHeap)
├─ small 尺寸變為 1
└─ 尚無平衡需要 (只有 1 元素)

狀態：
  small.data: [1] @ 0x2000 (size=1, capacity=4)
  large.data: [] @ 0x2100 (size=0, capacity=4)
  dh.smallSize = 1, dh.largeSize = 0
```

**Insert 3：**
```
操作：
├─ 將 3 添加到 small (MaxHeap)
├─ sift-up: 3 > 1, 交換 3 和 1
├─ 根節點現為 3 (最大值)
└─ small 尺寸變為 2

堆操作追蹤：
  sift-up(1):
    ├─ small.data[1] = 3
    ├─ parent_idx = (1-1)/2 = 0
    ├─ small.data[0] = 1
    ├─ 比較：3 > 1? 是 → 交換
    └─ 交換後：small.data[0] = 3, small.data[1] = 1

狀態：
  small.data: [3, 1] @ 0x2000 (size=2, capacity=4)
  large.data: [] @ 0x2100 (size=0, capacity=4)
  dh.smallSize = 2, dh.largeSize = 0

記憶體位址：
  small.data[0] = 3 @ 0x2000
  small.data[1] = 1 @ 0x2004
```

**Insert -1：**
```
操作：
├─ 將 -1 添加到 large (MinHeap)
├─ large 只有 1 元素，無須平衡
└─ large 尺寸變為 1

狀態：
  small.data: [3, 1] @ 0x2000 (size=2, capacity=4)
  large.data: [-1] @ 0x2100 (size=1, capacity=4)
  dh.smallSize = 2, dh.largeSize = 1

記憶體位址：
  large.data[0] = -1 @ 0x2100
```

**堆樹結構圖：**
```
小堆 (MaxHeap):          大堆 (MinHeap):
    3                        -1
   /
  1

堆驗證：
  small: max(3,1) = 3 ✓
  large: min(-1) = -1 ✓
  smallSize = 2 >= largeSize = 1 ✓
```

**中位數計算：**
```
smallSize = 2, largeSize = 1
中位數 = small.data[0] = 3 ✓
```

---

#### **步驟 2：視窗 [3, -1, -3] (索引 1-3)**

**移除 1，插入 -3：**

**Erase 1（使用延遲刪除）：**
```
操作：
├─ 在 HashMap 中記錄「1 已移除」
├─ delayed.entries[hash(1)] = {key:1, count:1, used:1}
├─ 不立即從堆中移除
└─ small.size 保持為 2（邏輯大小會在後續調整）

HashMap 更新：
  hash(1) = (1 * 31) % 4 = 31 % 4 = 3
  delayed.entries[3]: key=1, count=1, used=1 (@ 0x3030)

記憶體更新 (HashMap @ 0x3000)：
  0x3030 ┌─ key = 1 ──────┐
         ├─ count = 1 ────┤
         └─ used = 1 ─────┘
```

**Insert -3：**
```
操作：
├─ 比較 -3 與 small.data[0] = 3
├─ -3 < 3，所以添加到 large
├─ large.sift-up: -3 < -1? 是 → 交換
└─ large 根節點變為 -3

堆操作追蹤：
  large.sift-up(1):
    ├─ large.data[1] = -3
    ├─ parent_idx = 0
    ├─ large.data[0] = -1
    ├─ 比較：-3 < -1? 是 → 交換
    └─ 交換後：large.data[0] = -3, large.data[1] = -1

狀態：
  small.data: [3, 1] @ 0x2000 (size=2, logicalSize=1)
  large.data: [-3, -1] @ 0x2100 (size=2, capacity=4)
  dh.smallSize = 1 (已移除 1), dh.largeSize = 2

記憶體位址：
  large.data[0] = -3 @ 0x2100
  large.data[1] = -1 @ 0x2104
```

**堆樹結構圖：**
```
小堆 (MaxHeap):          大堆 (MinHeap):
    3*                        -3
   /                         /
  1*                        -1

* 表示在 delayed map 中標記為已移除
```

**平衡檢查：**
```
smallSize = 1 (邏輯), largeSize = 2
需要平衡：smallSize + 1 == largeSize
操作：
├─ large.data[0] (-3) 移動到 small
├─ small.sift-up(-3):
│   ├─ 位置在 small.data[1] = -3
│   ├─ 但須先移除堆頂 → 需要 pop_max
│   └─ 複雜操作...
```

實際平衡會涉及 pop 和 push 操作。為簡化，此步驟後：

**假設平衡完成後狀態：**
```
如果實現 rebalance 為 pop large top → push to small：
  small 會收到 -3
  small.sift-up(-3) 與現有 3, 1 比較

實際結果（取決於實現）可能為：
  small: [-3, 1] 或 [1, 3] 等
  large: [-1] 或更新的配置
```

**保守估計（延遲刪除生效時）：**
```
狀態：
  small.data: [3, 1] @ 0x2000 (size=2, actual=1 after lazy del)
  large.data: [-3, -1] @ 0x2100 (size=2)
  dh.smallSize 邏輯調整, dh.largeSize = 2

中位數候選：
  small 頂 = 3 (但 1 已標記刪除)
  large 頂 = -3
  → 若 smallSize 真實值 = 1, largeSize = 2
  → 中位數 = large.data[0] = -3 ✓
```

---

#### **步驟 3：視窗 [-1, -3, 5] (索引 2-4)**

**移除 3，插入 5：**

**Erase 3：**
```
操作：
├─ 在 HashMap 中記錄「3 已移除」
├─ hash(3) = (3 * 31) % 4 = 93 % 4 = 1
├─ delayed.entries[1] = {key:3, count:1, used:1}
└─ 不立即從堆中移除 (small 中的 3)

HashMap 更新 (@ 0x3000)：
  0x3010 ┌─ key = 3 ──────┐
         ├─ count = 1 ────┤
         └─ used = 1 ─────┘
```

**Insert 5：**
```
操作：
├─ 比較 5 與 small 頂
├─ 5 > small.data[0] = 3，添加到 small
├─ small.sift-up(5):
│   ├─ 插入位置：small.data[2] = 5
│   ├─ parent_idx = 0
│   ├─ small.data[0] = 3
│   ├─ 比較：5 > 3? 是 → 交換
│   └─ 交換後：small.data[0] = 5, small.data[2] = 3

狀態：
  small.data: [5, 1, 3] @ 0x2000 (size=3)
  記憶體位址：
    0x2000: [5]
    0x2004: [1]
    0x2008: [3]
```

**堆樹結構圖：**
```
小堆 (MaxHeap):
      5
     / \
    1   3*

大堆 (MinHeap):
     -3
     /
   -1
```

**平衡與清理：**
```
smallSize = 3, largeSize = 2
不平衡：smallSize > largeSize + 1
操作：
├─ pop small 頂 (5) → push to large
├─ large.sift-up(5):
│   └─ 5 成為新根或移動到適當位置
└─ small 尺寸減 1, large 尺寸加 1

同時進行延遲刪除清理：
├─ 若 small 堆頂是已刪除項，pop 並實際移除
├─ 此處 small 堆頂是 5，非已刪除
└─ 無法進行實時清理
```

**簡化後狀態：**
```
假設平衡完成：
  small.size = 2 (去掉 1 個 top)
  large.size = 3 (新增 1 個)

可能的堆配置：
  small: [3, 1] 或 [1, 3]
  large: [-3, 5, -1] 或排序後的配置
```

## 6. 記憶體存取模式比較

### 6.1 陣列式堆 vs 指標式樹

#### **陣列式堆（當前實現）**

**優點：**
```
┌─────────────────────────────────────────────────────────┐
│ 1. 高度快取局部性 (Cache Locality)                      │
│    ├─ 堆元素在連續記憶體中                              │
│    ├─ 父子節點位置由公式計算，無額外指標               │
│    └─ L1/L2 快取命中率高                                 │
├─────────────────────────────────────────────────────────┤
│ 2. 記憶體開銷小                                          │
│    ├─ 無需為每個節點儲存指標 (省 16 bytes/節點)        │
│    ├─ 只需儲存 int 值                                    │
│    └─ 堆高度為 O(log n)，指標開銷若有會很大             │
├─────────────────────────────────────────────────────────┤
│ 3. 動態擴展簡單                                          │
│    ├─ realloc 增加容量                                  │
│    ├─ 無需更新結構化指標                                │
│    └─ 批次操作效率高                                    │
└─────────────────────────────────────────────────────────┘
```

**缺點：**
```
┌─────────────────────────────────────────────────────────┐
│ 1. 插入/刪除需要 sift-up/sift-down                      │
│    ├─ O(log n) 時間複雜度                               │
│    ├─ 涉及多次陣列元素移動                              │
│    └─ 難以快速定位特定元素進行刪除                      │
├─────────────────────────────────────────────────────────┤
│ 2. 不適合頻繁隨機刪除                                   │
│    ├─ 必須搜尋元素 (O(n) 若無索引)                      │
│    └─ 本題用延遲刪除規避此問題                          │
└─────────────────────────────────────────────────────────┘
```

**記憶體開銷比較（n=1000 元素）：**
```
陣列式堆：
  ├─ int 陣列: 1000 * 4 bytes = 4 KB
  ├─ Heap 結構: 24 bytes
  └─ 總計: ~4 KB

指標式樹（假設左右子指標）：
  ├─ 樹節點: 1000 * (4 + 8 + 8) = 20 KB
  │         (int value + left ptr + right ptr)
  └─ 總計: ~20 KB

節省比例：75% (4 KB vs 20 KB)
```

#### **指標式樹**

**優點：**
```
┌─────────────────────────────────────────────────────────┐
│ 1. O(1) 指標存取                                        │
│    ├─ 快速跳轉到子節點                                  │
│    └─ 不需計算索引                                      │
├─────────────────────────────────────────────────────────┤
│ 2. 靈活刪除                                              │
│    ├─ 無需移動陣列元素                                  │
│    ├─ 直接斷開指標鏈接                                  │
│    └─ 易於實現 lazy deletion                            │
└─────────────────────────────────────────────────────────┘
```

**缺點：**
```
┌─────────────────────────────────────────────────────────┐
│ 1. 快取不友善                                            │
│    ├─ 節點分散在堆記憶體中                              │
│    ├─ 每個節點訪問可能觸發快取未命中                   │
│    └─ 性能最多下降 50% (取決於記憶體壓力)              │
├─────────────────────────────────────────────────────────┤
│ 2. 記憶體開銷大 (75% 額外)                             │
│    ├─ 每個節點需要 2 個指標 (16 bytes on 64-bit)      │
│    └─ 動態分配產生內存碎片                              │
└─────────────────────────────────────────────────────────┘
```

---

### 6.2 開放定址哈希表 (Open Addressing) vs 鏈式哈希表 (Chaining)

**當前實現：開放定址法 (Open Addressing)**

```
HashMap 結構：
  entries: HashEntry[容量]
  每個 slot 存 (key, count, used) 三元組

衝突解決：線性探測 (Linear Probing)
  hash(key) = (key * 31) % capacity
  衝突時 hash, hash+1, hash+2, ... (mod capacity)
```

#### **開放定址法優點**

```
┌─────────────────────────────────────────────────────────┐
│ 1. 記憶體局部性好                                        │
│    ├─ 所有項目連續儲存                                  │
│    └─ 存取相鄰 slot 時快取友善                          │
├─────────────────────────────────────────────────────────┤
│ 2. 無額外指標開銷                                        │
│    ├─ 只需 int + int + char + padding (16 bytes)      │
│    └─ vs 鏈式每個節點需指標 (24+ bytes)                │
├─────────────────────────────────────────────────────────┤
│ 3. 快取行利用率高                                        │
│    ├─ 64 bytes 快取行可容納 4 個 HashEntry             │
│    └─ 順序掃描時批次處理                                │
└─────────────────────────────────────────────────────────┘
```

#### **開放定址法缺點**

```
┌─────────────────────────────────────────────────────────┐
│ 1. 聚集現象 (Clustering)                                │
│    ├─ 高衝突率時線性探測形成聚集                       │
│    ├─ 探測長度增加，性能惡化                            │
│    └─ 本題 k=3 時衝突機率低，影響有限                   │
├─────────────────────────────────────────────────────────┤
│ 2. 刪除操作複雜                                          │
│    ├─ 需要「墓碑」標記 (used=0)                         │
│    └─ 無法真正移除，造成死區 (Dead Zone)               │
├─────────────────────────────────────────────────────────┤
│ 3. 負載因子限制                                          │
│    ├─ 通常要求 load factor < 0.7                       │
│    └─ 浪費記憶體空間                                    │
└─────────────────────────────────────────────────────────┘
```

#### **鏈式哈希表**

**結構：**
```
HashMap:
  buckets: LinkedList[容量]
  每個 bucket 儲存衝突項目的鏈表

衝突解決：鏈表連結
  hash(key) → bucket[i] → node1 → node2 → ...
```

**優點：**
```
├─ 刪除 O(1) (取得指標後直接移除)
├─ 無聚集問題
├─ 負載因子可達 1.0+
└─ 簡單直觀
```

**缺點：**
```
├─ 記憶體開銷：每個 node 需 8 bytes 指標 (64-bit)
├─ 快取不友善：node 分散堆記憶體
├─ 動態分配產生內存碎片
└─ 性能波動大 (取決於鏈長)
```

#### **本題選擇分析**

```
延遲刪除場景：
  ├─ 通常刪除操作集中在視窗移出時
  ├─ 同一元素不會連續刪除多次
  └─ 開放定址 + lazy deletion 足夠

負載估計：
  k = 3 時，延遲刪除項 ≤ 2*k = 6
  初始 capacity = 4，達到 6 時需要 rehash
  rehash 時 load factor = 6/8 = 75% (仍可接受)

性能對比：
  操作        │ 開放定址  │ 鏈式
  ────────────┼──────────┼──────
  查詢 hit    │ O(1) avg │ O(1)
  插入        │ O(1)     │ O(1)
  刪除 (lazy) │ O(1)     │ O(1)
  快取性能    │ 優 (局部)│ 差 (分散)
  記憶體      │ 優 (小)  │ 差 (大)
```

## 7. 堆疊記憶體對比

### 7.1 迭代式堆操作

**sift-up 迭代實現 (當前)：**
```c
void sift_up(Heap* h, int idx) {
    while (idx > 0) {
        int parent = (idx - 1) / 2;
        if (compare(h->data[idx], h->data[parent])) {
            swap(&h->data[idx], &h->data[parent]);
            idx = parent;
        } else {
            break;
        }
    }
}
```

**堆疊使用：**
```
函數呼叫堆疊：
┌─────────────────────────────────┐
│ 返回位址                         │  8 bytes (64-bit)
├─────────────────────────────────┤
│ Heap* h (參數)                   │  8 bytes
├─────────────────────────────────┤
│ int idx (參數)                   │  4 bytes
├─────────────────────────────────┤
│ int parent (局部)                │  4 bytes
├─────────────────────────────────┤
│ (padding)                        │  4 bytes
├─────────────────────────────────┤
│ 迴圈計數/狀態                     │  0 bytes (暫存器優化)
└─────────────────────────────────┘

總堆疊框架：28 bytes (含 padding)
迴圈深度：O(log n) 次反覆，但無遞迴堆疊增長
最大堆疊用量：28 bytes (常數)
```

**最壞情況（n = 1000000）：**
```
sift-up 反覆次數：log₂(1000000) ≈ 20 次
每次迴圈只是變數更新，無額外堆疊分配
總堆疊使用：28 bytes (固定)
```

### 7.2 遞迴式堆操作

**sift-up 遞迴實現（假設）：**
```c
void sift_up_recursive(Heap* h, int idx) {
    if (idx == 0) return;

    int parent = (idx - 1) / 2;
    if (compare(h->data[idx], h->data[parent])) {
        swap(&h->data[idx], &h->data[parent]);
        sift_up_recursive(h, parent);  // 遞迴呼叫
    }
}
```

**堆疊框架結構：**
```
遞迴深度 d 時的呼叫堆疊：

第 0 層（最新呼叫）：
┌─────────────────────────┐
│ 返回位址 (8B)           │
│ Heap* h (8B)            │
│ int idx (4B)            │
│ int parent (4B)         │
│ padding (4B)            │
├─────────────────────────┤ 28 bytes per frame
│ 第 1 層呼叫框架          │
├─────────────────────────┤ 28 bytes per frame
│ 第 2 層呼叫框架          │
├─────────────────────────┤
│ ...                     │
├─────────────────────────┤
│ 第 d-1 層呼叫框架        │
└─────────────────────────┘
                總: d * 28 bytes
```

**最壞情況堆疊成長：**

```
n = 1000000 (100 萬元素)

迭代實現：
  ├─ 堆深度：log₂(1000000) = 20
  ├─ 堆疊框架數：1
  ├─ 總堆疊用量：28 bytes
  └─ 堆疊溢出風險：無

遞迴實現：
  ├─ 遞迴深度：20 (same as iteration count)
  ├─ 堆疊框架數：20
  ├─ 總堆疊用量：20 * 28 = 560 bytes
  └─ 堆疊溢出風險：低但存在

堆疊對比（百萬級元素）：
  操作      │ 迭代  │ 遞迴      │ 比率
  ──────────┼───────┼──────────┼──────
  堆疊深度  │ 0     │ 20 frames│ 20倍
  單幀大小  │ 28B   │ 28B      │ 1倍
  總堆疊用  │ 28B   │ 560B     │ 20倍
  最大框架  │ 1     │ 20       │ 20倍
```

**遞迴風險分析：**
```
典型堆疊大小：8 MB (現代系統)

遞迴深度 d 時堆疊用量：d * 28 bytes

d = 1000 時：28 KB（安全）
d = 100000 時：2.8 MB（接近限制）
d = 285714 時：8 MB（堆疊溢出）

本堆算法：
  最大遞迴深度 ≈ log₂(1000000) ≈ 20
  遞迴實現堆疊用量 ≈ 560 bytes（非常安全）

結論：遞迴實現對本題安全，但迭代更優（無開銷）
```

## 8. 退化情況分析

### 8.1 延遲刪除導致的堆膨脹

**場景：嚴格遞減序列**

```
nums = [100, 99, 98, 97, 96, ..., 1], k = 3
```

**初始化和前三個視窗：**
```
視窗 1: [100, 99, 98]
  small: [100, 99, 98]
  large: []

視窗 2: [99, 98, 97]
  erase(100): delayed[100] = 1 ← 延遲刪除！
              small 堆仍包含 100
  insert(97)

視窗 3: [98, 97, 96]
  erase(99): delayed[99] = 1 ← 又一個！
              small 堆仍包含 99
  insert(96)
```

**堆膨脹過程：**
```
操作序列（視窗滑動）：
  erase(100) → erase(99) → erase(98) → erase(97) → ...

每次 erase 都標記一個元素為「延遲刪除」
但新元素被添加到 small 堆，old 元素不被移除

結果：
  small.size 邏輯遞減（記錄的邏輯大小）
  small 陣列物理容量增長 → O(n) 不必要的空間
```

**堆陣列成長曲線：**
```
時間軸 (視窗移動):  0    1    2    3    4    5   ...  99   100
                  ┌────────────────────────────────────┐
small.data 大小:  │3    4    5    6    7    8   ...  100  100
                  │╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱╱
                  └────────────────────────────────────┘
                 延遲刪除堆積

預期最優大小: 3 (總是 k 元素)
實際大小: 100 (所有過期元素)

空間浪費比例: (100-3)/100 ≈ 97%
```

**記憶體配置演變：**

**初始狀態（視窗 0-2）：**
```
small.data @ 0x2000:
  ┌────────┬────────┬────────┬────────┐
  │ 100    │ 99     │ 98     │ (空)   │
  └────────┴────────┴────────┴────────┘
  size = 3, capacity = 4
```

**視窗 3-5 後：**
```
small.data @ 0x2000 (realloc 到 0x2100):
  ┌──────┬──────┬──────┬──────┬──────┬──────┬──────┬──────┐
  │ 98*  │ 97*  │ 96   │ 95   │ 94   │ (空) │ ...  │ (空) │
  └──────┴──────┴──────┴──────┴──────┴──────┴──────┴──────┘
  size = 5, capacity = 8
  (* 標記為延遲刪除)

delayed HashMap 同步膨脹：
  {100:1, 99:1, 98:1, 97:1}
  增長到 capacity = 16
```

**視窗結束時（100 個元素，k=3）：**
```
small.data @ 0x2100 (realloc 到 0x3000):
  ┌────┬────┬────┬────┬────┬ ... ┬────┬────┬────┬────┐
  │ 3* │ 2* │ 1* │ 0* │ ... (全部延遲刪除) ... │
  └────┴────┴────┴────┴────┴ ... ┴────┴────┴────┴────┘
  size = 100, capacity = 128
  (* 幾乎全部項目都標記為已刪除)

有效元素個數: 3 (最後的視窗)
浪費的槽位: 125 (97.6% 浪費)

記憶體用量：
  small.data: 128 * 4 bytes = 512 bytes
  delayed HashMap: 128 個 entry * 16 bytes = 2048 bytes
  總計: 2560 bytes (用於 3 個實際元素！)

vs 最優方案:
  3 個元素: ~100 bytes

比例: 25.6倍 放大！
```

### 8.2 堆膨脹對性能的影響

**插入操作性能惡化：**

```
n = 100 個元素，k = 3

單次 insert 成本分析：
  最優情況: O(log k) = O(log 3) ≈ 1-2 步

延遲刪除後:
  small.size 實際 = 100 (雖然邏輯 = 3)
  sift-up: O(log 100) ≈ 7 步（增加 3-5 倍！）

記憶體存取：
  最優: L1/L2 快取命中（堆小且熱）
  延遲刪除: 快取未命中率增加（堆大且冷）
```

**延遲刪除清理的必要性：**

```
清理策略 1: 被動清理 (On-access)
  ├─ 每次 pop 堆頂時檢查是否延遲刪除
  ├─ 若是則真正移除，再 pop 下一個
  └─ 成本: O(log n) per removal

清理策略 2: 主動清理 (On-demand)
  ├─ 當延遲刪除項達到某閾值時重建堆
  ├─ e.g., delayed.size > small.size / 2
  └─ 成本: O(n) 重建，但攤銷後 O(log n)

本題應使用策略 1：
  ├─ 視窗只涉及最多 2k 個元素
  ├─ 延遲刪除項永不超過 k
  ├─ 堆膨脹風險低
  └─ 但在惡劣輸入下仍需注意
```

## 9. 結論與優化建議

### 9.1 本實現的設計選擇

```
├─ 陣列式堆：快取友善，適合連續操作
├─ 延遲刪除：規避 O(n) 搜尋開銷
├─ 開放定址 HashMap：記憶體緊湊，適合小容量
└─ 迭代 sift-up/down：無堆疊溢出風險
```

**適用場景：**
```
✓ k 較小 (k < 100)：延遲項有限，堆膨脹不明顯
✓ 輸入接近隨機：無特殊聚集，性能穩定
✓ 無特殊大元素集中移除：避免延遲刪除累積
```

**不適用場景：**
```
✗ k 很大 (k > 1000)：延遲項累積，空間浪費
✗ 輸入單調遞減/遞增：所有過期元素延遲刪除
✗ 對空間敏感的嵌入式系統：128 倍開銷不可接受
```

### 9.2 優化建議

**1. 主動清理機制：**
```c
// 當延遲刪除項過多時，重建堆
if (delayed.size > small.size / 2) {
    rebuild_small_heap();  // O(n log n)
    clear_delayed_map();
}
```

**2. 迫切清理 (Eager cleanup)：**
```c
// pop 時主動清理堆頂
int pop_and_clean(Heap* h) {
    while (h->size > 0 && is_deleted(h->data[0])) {
        // 真正刪除堆頂
        h->data[0] = h->data[--h->size];
        sift_down(h, 0);
    }
    if (h->size > 0) return h->data[0];
    return NOT_FOUND;
}
```

**3. 雙堆平衡優化：**
```c
// 記錄實際有效元素數，提前觸發清理
int effective_small = small.size - delayed_in_small_count;
if (effective_small < k/2) {
    // 觸發強制清理
    purge_delayed_items(small);
}
```
