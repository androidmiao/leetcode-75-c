# LeetCode 1768 - Merge Strings Alternately 記憶體分析

## 1. Struct 記憶體佈局

本題使用 `char*` 字符指針而非結構體，三個字符陣列的記憶體佈局如下：

### word1 陣列佈局 (假設起始於 0xA000)
```
word1: "abc\0"

位址          內容      大小
0xA000        'a'       1 byte
0xA001        'b'       1 byte
0xA002        'c'       1 byte
0xA003        '\0'      1 byte (null terminator)
─────────────────────────────
總計: 4 bytes (strlen(word1) = 3)
```

### word2 陣列佈局 (假設起始於 0xB000)
```
word2: "pqr\0"

位址          內容      大小
0xB000        'p'       1 byte
0xB001        'q'       1 byte
0xB002        'r'       1 byte
0xB003        '\0'      1 byte (null terminator)
─────────────────────────────
總計: 4 bytes (strlen(word2) = 3)
```

### res 陣列佈局 (假設起始於 0xC000)
```
res: 動態分配 (malloc)

位址          初值      大小
0xC000        ?         1 byte
0xC001        ?         1 byte
0xC002        ?         1 byte
0xC003        ?         1 byte
0xC004        ?         1 byte
0xC005        ?         1 byte
0xC006        '\0'      1 byte (null terminator)
─────────────────────────────
總計: 7 bytes (n1 + n2 + 1 = 3 + 3 + 1)
```

## 3. 陣列索引版對應

### 索引與記憶體位址映射

```
word1[i] 的記憶體位址 = 0xA000 + i
word2[j] 的記憶體位址 = 0xB000 + j
res[k]   的記憶體位址 = 0xC000 + k
```

### 範例映射表 (word1="abc", word2="pqr")

| 操作順序 | i  | j  | k  | 來源     | word1[i]/word2[j] | res[k] | 位址           |
|---------|----|----|----|---------|--------------------|--------|----------------|
| 初始    | 0  | 0  | 0  | -       | -                  | -      | -              |
| 1       | 0  | 0  | 0  | word1   | 'a'                | 'a'    | 0xC000         |
|         | 1  | 0  | 1  | (移動後)|                    |        |                |
| 2       | 1  | 0  | 1  | word2   | 'p'                | 'p'    | 0xC001         |
|         | 1  | 1  | 2  | (移動後)|                    |        |                |
| 3       | 1  | 1  | 2  | word1   | 'b'                | 'b'    | 0xC002         |
|         | 2  | 1  | 3  | (移動後)|                    |        |                |
| 4       | 2  | 1  | 3  | word2   | 'q'                | 'q'    | 0xC003         |
|         | 2  | 2  | 4  | (移動後)|                    |        |                |
| 5       | 2  | 2  | 4  | word1   | 'c'                | 'c'    | 0xC004         |
|         | 3  | 2  | 5  | (移動後)|                    |        |                |
| 6       | 3  | 2  | 5  | word2   | 'r'                | 'r'    | 0xC005         |
|         | 3  | 3  | 6  | (移動後)|                    |        |                |

### 複製操作說明

```
C 語言複製操作:
res[k] = word1[i];  // 當 i < n1 時
res[k] = word2[j];  // 當 j < n2 時

記憶體層級操作:
*(res + k) = *(word1 + i);
*(res + k) = *(word2 + j);

指針算術:
word1 是 char* 指向 0xA000
word1[i] ≡ *(word1 + i) ≡ *(0xA000 + i)

相同原理適用於 word2 和 res
```

## 6. 記憶體存取模式比較

### 堆棧分配 vs 堆分配

本題採用**堆分配** (malloc)，因為輸出字符串長度在運行時才知道。

| 特性           | 堆棧分配 (char[])        | 堆分配 (malloc)           |
|----------------|----------------------|----------------------|
| **分配時機**   | 編譯時 (固定大小)     | 運行時 (動態大小)    |
| **分配速度**   | 非常快 (只修改 SP)    | 較慢 (遍歷空閒塊)    |
| **解放速度**   | 自動 (函數返回時)     | 手動 (free 調用)     |
| **大小限制**   | 堆棧大小限制          | 系統可用內存         |
| **緩存特性**   | 優秀 (局部性)         | 良好 (需要 malloc)   |
| **碎片化風險** | 無 (線性堆棧)         | 有 (堆碎片)          |
| **適用場景**   | 大小已知、小型數據    | 大小未知、大型數據   |

### 本題的選擇理由

```
輸出長度 = n1 + n2
           = strlen(word1) + strlen(word2)

→ 只有在調用 strlen() 後才能確定
→ strlen() 在運行時執行
→ 因此必須使用 malloc (堆分配)
→ 堆棧分配不可行
```

### 記憶體存取效率分析

```
【堆棧訪問】(如果可行)
- 訪問延遲: ~4-10 CPU 周期
- L1 快取命中: 高概率
- 局部性: 最優 (連續棧內存)

【堆訪問】(本題的情況)
- 訪問延遲: ~10-40 CPU 周期 (含快取未命中)
- L1 快取命中: 中等
- 局部性: 取決於碎片化程度

【本題優化點】
res 是連續分配，所有訪問呈線性模式:
res[0], res[1], res[2], ... → 高度局部性
word1 和 word2 的訪問也是線性的
→ 整體快取命中率較高
```

### 碎片化比較

```
【堆棧】
分配: [word1 4B][word2 4B][padding][其他]
解放: 自動全部釋放
碎片化: 0%

【堆】(本題)
首次分配: malloc(7) → 0xC000
第一次調用: [res 7B][空閒1][空閒2]...
碎片化: 依賴於 malloc 實現和其他分配模式

常見 malloc 實現 (如 jemalloc):
- 使用 bin 系統减少碎片化
- 本題單次分配，碎片化風險低
```

## 7. 堆疊記憶體對比

### 迭代方法 (當前實現)

#### 堆棧幀結構

```
函數調用堆棧:
┌────────────────────────────────────────┐
│ main() 的堆棧幀                        │
├────────────────────────────────────────┤
│ (main 的局部變數、返回地址)            │
└────────────────────────────────────────┘
                   ↓ 函數調用
┌────────────────────────────────────────┐
│ mergeAlternately() 的堆棧幀            │
├────────────────────────────────────────┤
│ 返回地址 (RIP)           │ 0x7FFF0100  │
│ 舊 RBP                   │ 0x7FFF0108  │
│ word1 參數              │ 0x7FFF0110  │ ← 指向 0xA000
│ word2 參數              │ 0x7FFF0118  │ ← 指向 0xB000
│ returnSize 參數         │ 0x7FFF0120  │
│ n1 (局部)               │ 0x7FFF0128  │ = 3
│ n2 (局部)               │ 0x7FFF0130  │ = 3
│ res (局部)              │ 0x7FFF0138  │ ← 指向 0xC000
│ i (局部)                │ 0x7FFF0140  │ = 0..n1
│ j (局部)                │ 0x7FFF0148  │ = 0..n2
│ k (局部)                │ 0x7FFF0150  │ = 0..n1+n2
└────────────────────────────────────────┘
```

#### 迭代執行流程

```
【初始化】
堆棧分配:
- n1, n2: 2 × sizeof(size_t) = 16 bytes (64-bit)
- res: 1 × sizeof(char*) = 8 bytes
- i, j, k: 3 × sizeof(size_t) = 24 bytes
- 合計: 48 bytes (含 padding)

堆分配:
- malloc(7) → 7 bytes 於 0xC000

【迴圈執行】
內存操作:
- 每次迭代: 6 次棧變數讀取 (i, j, k, n1, n2)
- 每次迭代: 1 次堆讀取 (word1[i] 或 word2[j])
- 每次迭代: 1 次堆寫入 (res[k])
- 每次迭代: 3 次棧寫入 (更新 i/j/k)

【函數退出】
堆棧自動釋放 (48 bytes)
返回 res 指針 (0xC000) 給調用者
堆內存由調用者負責 free
```

#### 時間複雜度

```
初始化: O(1)
迴圈:   O(n1 + n2)
清理:   O(1)
總計:   O(n1 + n2)
```

### 假設的遞歸方法

#### 遞歸函數簽名

```c
// 假設的遞歸實現
char *mergeAlternately_recursive(
    char *word1, char *word2,
    size_t i, size_t j, size_t k,
    char *res, size_t n1, size_t n2
) {
    // 終止條件
    if (i >= n1 && j >= n2) {
        res[k] = '\0';
        return res;
    }

    // 遞歸: 字符來自 word1
    if (i < n1) {
        res[k] = word1[i];
        return mergeAlternately_recursive(
            word1, word2, i+1, j, k+1, res, n1, n2
        );
    }

    // 遞歸: 字符來自 word2
    if (j < n2) {
        res[k] = word2[j];
        return mergeAlternately_recursive(
            word1, word2, i, j+1, k+1, res, n1, n2
        );
    }

    return res;
}
```

#### 遞歸堆棧幀

```

│ 返回地址                       │
│ 舊 RBP                         │
│ word1, word2, res, n1, n2      │

│ (其他局部變數)                 │ ← 堆棧: ~100 bytes
└────────────────────────────────┘
         ↓ 遞歸調用
【調用 2】 mergeAlternately_recursive(..., i=1, j=0, k=1)
┌────────────────────────────────┐
│ 返回地址                       │
│ 舊 RBP                         │
│ word1, word2, res, n1, n2      │
│ i=1, j=0, k=1                  │
│ (其他局部變數)                 │ ← 堆棧: ~100 bytes
└────────────────────────────────┘
         ↓ 遞歸調用
【調用 3】 mergeAlternately_recursive(..., i=1, j=1, k=2)
┌────────────────────────────────┐
│ ...                            │
└────────────────────────────────┘
         ↓ ... (共 6 次遞歸調用)
【調用 7】 (終止條件: i=3, j=3)
┌────────────────────────────────┐

         ↑ 返回

         ↑ 返回

         ... (回收堆棧幀)
【調用 1】 返回 res 至 main

堆棧最深度: 7 × 100 bytes ≈ 700 bytes (取決於編譯器優化)
```

#### 遞歸 vs 迭代 對比

| 指標           | 迭代 (當前)          | 遞歸 (假設)        |
|----------------|---------------------|-------------------|
| **堆棧深度**   | 1 層                | O(n1+n2) 層       |
| **堆棧空間**   | ~48 bytes           | ~700 bytes (6次)  |
| **調用次數**   | 1 次                | 6 次 (Example 1)  |
| **函數開銷**   | 無 (無遞歸開銷)     | 6 × ~20 bytes     |
| **快取效率**   | 優秀                | 良好 (但多層)    |
| **代碼複雜度** | 簡單                | 複雜              |
| **易於理解**   | 清晰                | 難懂              |
| **性能**       | 快 (~6 迭代)        | 慢 (~6 次調用)   |
| **棧溢出風險** | 無                  | 中等 (深度遞歸)   |

#### 遞歸的缺點

```
【堆棧爆炸】
如果 n1 + n2 很大:
- word1 = 1000 字符
- word2 = 1000 字符
- 需要 2000 層遞歸
- 堆棧空間: 2000 × 100 ≈ 200 KB
- 典型堆棧大小: 1-8 MB
- 風險: 棧溢出!

【函數調用開銷】
每次調用 ~20-50 CPU 周期:
- 2000 次調用 × 30 周期 = 60,000 周期
- 迭代: 2000 次循環 × 5 周期 = 10,000 周期
- 遞歸性能: 下降 6 倍!

【尾遞歸優化】
一些編譯器 (如 GCC -O2) 可優化為迭代:
gcc -O2: 遞歸 → 編譯為迭代代碼
gcc -O0: 遞歸 → 真正的堆棧調用
```

#### 尾遞歸優化

```c
// 尾遞歸形式 (可被編譯器優化)
char *mergeAlternately_tail_recursive(
    char *word1, char *word2,
    size_t i, size_t j, size_t k,
    char *res, size_t n1, size_t n2
) {
    // 終止條件
    if (i >= n1 && j >= n2) {
        res[k] = '\0';
        return res;  // 尾部返回 (遞歸調用在最後)
    }

    if (i < n1) {
        res[k] = word1[i];
        // 這是尾遞歸: 最後一條語句是遞歸調用
        return mergeAlternately_tail_recursive(
            word1, word2, i+1, j, k+1, res, n1, n2
        );
    }

    if (j < n2) {
        res[k] = word2[j];
        // 這也是尾遞歸
        return mergeAlternately_tail_recursive(
            word1, word2, i, j+1, k+1, res, n1, n2
        );
    }

    return res;
}

// GCC -O2 優化後,編譯器會將其轉換為:
// → 循環 (跳轉而非調用)
// → 堆棧深度: 1 (常數,不隨遞歸深度增加)
// → 性能: 與迭代相同
```

### 設計結論

#### 為何選擇迭代

```
【迭代設計優勢】
✓ 簡單直觀: 一個 while 迴圈,易於理解
✓ 高效: 無函數調用開銷
✓ 安全: 無堆棧溢出風險
✓ 通用: 工作於所有編譯器 (無優化依賴)

【遞歸設計劣勢】
✗ 複雜: 需要傳遞多個參數
✗ 低效: 函數調用開銷累積
✗ 危險: 大字符串導致堆棧溢出
✗ 不必要: 遞歸無特殊優勢 (無重疊子問題)

【選擇】
→ 迭代是最佳方案
→ 簡單、高效、安全
```

## 結論

本 LeetCode 1768 題的記憶體分析顯示:

1. **記憶體佈局**: 三個 char 陣列 (word1, word2, res) 分別位於堆上,由指針指向

6. **堆分配**: malloc(n1+n2+1),由調用者負責 free

此設計體現了 C 語言的高效性和低級控制能力。
