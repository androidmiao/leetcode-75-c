# LeetCode 141 - Linked List Cycle 記憶體分析

## 1. Struct 記憶體佈局

### `struct ListNode` 結構定義
```c
struct ListNode {
    int val;              // 4 bytes @ offset 0
    struct ListNode *next; // 8 bytes @ offset 8 (含4 bytes padding)
};
// 總計：16 bytes (含padding)
```

### 單一節點記憶體圖

假設節點位置在 `0xA000`：

```
┌───────────────────────────────┐
│      ListNode @ 0xA000        │
├───────────────┬───────────────┤
│  offset: 0-3  │  offset: 4-7  │
│    val: 3     │    [padding]  │
│  (4 bytes)    │   (4 bytes)   │
├───────────────┼───────────────┤
│  offset: 8-15                 │
│    next: 0xA010 ───▶ (指向下一節點) │
│   (8 bytes pointer)           │
└───────────────────────────────┘
```

### 記憶體配置表

| 欄位         | 類型              | 大小  | 偏移量 | 說明           |
|-------------|------------------|------|--------|---------------|
| `val`       | `int`            | 4B   | 0      | 節點數值       |
| (padding)   | —                | 4B   | 4      | 對齊用填充     |
| `next`      | `ListNode*`      | 8B   | 8      | 指向下一節點指標 |
| **總計**    | —                | **16B** | —     | 一個節點大小   |

## 3. 搜尋／操作過程追蹤

### Floyd 循環偵測演算法

演算法使用雙指標（slow 和 fast）：
- **slow**：每次移動 1 步
- **fast**：每次移動 2 步
- 若存在循環，fast 最終會趕上 slow

---

### 範例 1：head = [3,2,0,-4], pos = 1（存在循環）

#### 詳細追蹤

```
步驟 0 (初始化)
  slow = 0xA000 (node0, val=3)
  fast = 0xA000 (node0, val=3)
  slow == fast? YES (都在起點)
  ❌ 但此為初始狀態，繼續

步驟 1
  slow = slow.next = 0xA010 (node1, val=2)
  fast = fast.next.next = 0xA020 (node2, val=0)
  slow == fast? NO
  ├─ slow @ 0xA010, val=2
  └─ fast @ 0xA020, val=0
  繼續移動

步驟 2
  slow = slow.next = 0xA020 (node2, val=0)
  fast = fast.next.next = 0xA010 (node3->next = node1, val=2)
  slow == fast? NO
  ├─ slow @ 0xA020, val=0
  └─ fast @ 0xA010, val=2
  繼續移動

步驟 3
  slow = slow.next = 0xA030 (node3, val=-4)
  fast = fast.next.next = 0xA020 (node2, val=0)
  slow == fast? NO
  ├─ slow @ 0xA030, val=-4
  └─ fast @ 0xA020, val=0
  繼續移動

步驟 4
  slow = slow.next = 0xA010 (node1, val=2)
  fast = fast.next.next = 0xA010 (node1, val=2)
  slow == fast? YES ✓
  🎯 偵測到循環！
```

#### 指標移動視覺化

```
步驟 0:  head ─▶ S,F[3@A000] ─▶ [2@A010] ─▶ [0@A020] ─▶ [-4@A030]
                                                            ▲
                                                            └─────┘

步驟 1:  [3] ─▶ S[2@A010] ───▶ F[0@A020] ─▶ [-4] ─▶ [2]
                 ▲                                    ▲
                 │                                    └────┘

步驟 2:  [3] ─▶ [2] ───▶ S[0@A020] ─▶ [-4] ─▶ F[2@A010]
                         ▲                      ▲
                         │                      └────┘

步驟 3:  [3] ─▶ [2] ───▶ [0] ───▶ S[-4@A030] ─▶ F[2@A010]
                                  ▲                ▲
                                  │                └────┘

步驟 4:  [3] ─▶ [2] ───▶ [0] ───▶ [-4] ─▶ S,F[2@A010] ✓ 相遇！
                                           ▲
                                           └────┘
```

#### 指標變數追蹤表

| 步驟 | slow 位址  | slow 值 | fast 位址  | fast 值 | slow==fast? | 動作           |
|-----|-----------|--------|-----------|--------|------------|----------------|
| 0   | 0xA000    | 3      | 0xA000    | 3      | YES        | 初始化（跳過） |
| 1   | 0xA010    | 2      | 0xA020    | 0      | NO         | 繼續移動       |
| 2   | 0xA020    | 0      | 0xA010    | 2      | NO         | 繼續移動       |
| 3   | 0xA030    | -4     | 0xA020    | 0      | NO         | 繼續移動       |
| 4   | 0xA010    | 2      | 0xA010    | 2      | **YES**    | **循環偵測！** |

**結果**：存在循環 ✓

## 5. 記憶體存取模式比較

### 連結串列指標追蹤 vs 陣列基礎循環偵測

#### 連結串列指標追蹤（Floyd 演算法）

**優點**：
- **空間複雜度 O(1)**：僅使用兩個指標變數，無須額外資料結構
- **時間複雜度 O(n)**：最多遍歷 n + 循環長度 個節點
- **就地檢測**：無須複製或修改結構

**缺點**：
- **指標解參考開銷**：每次比較需解參考指標以獲取 `val` 和 `next`
- **隨機記憶體存取**：在非連續堆積記憶體上訪問節點，CPU 快取友善度低
- **多級間接性**：slow.next.next 涉及多次記憶體查詢，指令延遲高

#### 記憶體存取模式

```
連結串列指標追蹤（Floyd）:
  每次迭代:
    ├─ 載入 slow 指標值
    ├─ 載入 slow->next 位址
    ├─ 解參考 slow->next 取得下一節點
    ├─ 載入 fast 指標值
    ├─ 載入 fast->next 位址
    ├─ 解參考 fast->next 取得中間節點
    ├─ 載入 fast->next->next 位址
    ├─ 解參考 fast->next->next 取得下下個節點
    └─ 比較指標相等性 (slow == fast)

  記憶體模式: 隨機 (可能跳躍到堆積任意位置)
  快取命中率: 低（連結串列通常分散在記憶體中）
```

#### 指標大小與對齐

```
64 位元系統:
  - 單個指標: 8 bytes
  - 節點大小: 16 bytes (含 padding)
  - 指標封存成本: 50% 額外開銷

  每迭代記憶體讀取估算:
    - slow, fast 變數: 16 bytes (2 x 8)
    - 解參考 slow->next: 8 bytes (位址) + 16 bytes (node)
    - 解參考 fast->next->next: 8 + 16 + 8 + 16 bytes
    - 合計: ~88 bytes 記憶體流量 (包括快取未命中)
```

#### CPU 快取行為

```
最佳情況 (快取友善):
  多個節點連續配置在同一快取行 (64 bytes):
  [node0: 16B][node1: 16B][node2: 16B][node3: 16B]
  └─ 快取命中率: 75% ~ 90%

最壞情況 (快取不友善):
  節點分散在不同快取行:
  [node0][......][node1][......][node2][......][node3]
  └─ 快取命中率: 10% ~ 30%
  └─ 效能影響: L1/L2 快取未命中導致 stall
```

#### 搜尋複雜度比較表

| 方法              | 時間複雜度 | 空間複雜度 | 記憶體存取 | 快取友善度 | 適用情景              |
|------------------|-----------|-----------|---------|---------|---------------------|
| Floyd (指標追蹤)  | O(n)      | O(1)      | O(n)    | 低      | 記憶體受限, 無額外空間 |
| HashSet (O(1))   | O(n)      | O(n)      | O(n)    | 中      | 快速存取, 充足記憶體  |
| 標記訪問過節點    | O(n)      | O(1)*     | O(n)    | 中      | 允許結構修改          |

## 6. 堆疊記憶體對比

### Floyd 循環偵測演算法堆疊分析

#### 遞歸 vs 迭代

**本實現（迭代）**：
```c
bool hasCycle(struct ListNode *head) {
    if (head == NULL || head->next == NULL) return false;

    struct ListNode *slow = head;  // 8 bytes (棧上)
    struct ListNode *fast = head;  // 8 bytes (棧上)

    while (fast != NULL && fast->next != NULL) {
        slow = slow->next;
        fast = fast->next->next;

        if (slow == fast) return true;
    }

    return false;
}

棧幀大小: 16 bytes (兩個指標)
遞歸深度: 0 (無遞歸)
額外分配: 無
棧增長: 常數 O(1)
```

#### 與遞歸實現的對比

```
遞歸循環偵測 (假設實現):
bool hasCycle_recursive(struct ListNode *slow, struct ListNode *fast) {
    if (fast == NULL || fast->next == NULL) return false;
    if (slow == fast) return true;

    slow = slow->next;                    // 遞歸調用前準備
    fast = fast->next->next;              // 同上

    return hasCycle_recursive(slow, fast); // 遞歸呼叫
}

遞歸棧幀大小: 24 bytes (slow, fast, return addr, alignment)
最大遞歸深度: O(n) (最壞情況無循環時)
總棧使用: O(n) * 24 bytes = O(n)
棧溢出風險: 高 (長串列)
```

#### 記憶體配置對比表

| 項目            | 迭代 (Floyd)  | 遞歸實現       |
|----------------|--------------|--------------|
| **棧幀大小**   | 16 bytes     | ~24 bytes    |
| **最大深度**   | 1 (常數)     | O(n)         |
| **總棧用量**   | O(1)         | O(n)         |
| **全域變數**   | 無           | 無           |
| **堆積分配**   | 無           | 無           |
| **快取局部性** | 好 (變數重用)| 差 (棧幀分散)|
| **呼叫開銷**   | 無           | O(n) 呼叫   |
| **棧溢出風險** | 無           | 高           |

#### 棧記憶體視覺化

```
迭代版本 (Floyd):

    │   main 函數      │

    │ │ hasCycle 呼叫    │

    │ │ ┌──────────────┐ │
    │ │ │ slow: 0xA000 │ │ 8 bytes
    │ │ ├──────────────┤ │
    │ │ │ fast: 0xA000 │ │ 8 bytes
    │ │ └──────────────┘ │
    │ └────────────────┬──┘
    │                  │

    │   main 函數         │
    │ ┌───────────────┐   │
    │ │ 呼叫 0        │   │
    │ │ 棧幀 (24B)    │   │
    │ │ slow=A000     │   │

    │ ├───────────────┤   │
    │ │ 呼叫 1        │   │
```

### 結論

Floyd 循環偵測演算法使用**迭代而非遞歸**，因此：
- **棧使用量**：常數 O(1)（僅 16 bytes）
- **無棧溢出風險**：適用於任意長度的連結串列
- **無函數呼叫開銷**：無遞歸返回位址、參數傳遞等額外成本
- **高效率**：純粹指標比較迴圈，無複雜的遞歸管理

## 演算法總結

### Floyd 循環偵測演算法特性

```
時間複雜度: O(n)
  - 無循環: 遍歷整個串列 n 個節點
  - 有循環: 遍歷 n + 循環檢測步數 (通常 < n)

空間複雜度: O(1)
  - 僅使用常數個指標變數
  - 無遞歸棧
  - 無輔助資料結構

核心原理:
  1. slow, fast 同時從 head 開始
  2. slow 每次移動 1 步，fast 每次移動 2 步
  3. 若存在循環，fast 必定追上 slow（因為 fast 速度是 slow 兩倍）
  4. 相等檢查 (slow == fast) 偵測循環
  5. fast 到達 NULL 表示無循環

記憶體存取特徵:
  - 指標追蹤: 隨機堆積記憶體存取
  - 快取行為: 通常不友善（節點分散）
  - 指標解參考: 每迭代多次
  - 總傳輸: ~88 bytes/迭代（包括快取未命中）

最佳應用:
  ✓ 記憶體受限環境 (O(1) 空間)
  ✓ 無法修改原始結構 (不需標記訪問)
  ✓ 不允許額外資料結構
  ✗ 對快取友善度要求高 (指標追蹤低效)
```
