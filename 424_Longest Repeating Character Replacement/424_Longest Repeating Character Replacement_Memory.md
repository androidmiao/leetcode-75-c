# LeetCode 424 - 最長重複字元替換 (Longest Repeating Character Replacement) 記憶體分析

## 1. 堆疊記憶體佈局

### 函數呼叫時的堆疊幀結構

```
堆疊記憶體配置 (高位址至低位址):
┌─────────────────────────────────────┐
│     返回位址 & 舊 RBP                │ ← ESP + 12 (進入函數)
├─────────────────────────────────────┤
│     參數: int k                     │ ← ESP + 8
├─────────────────────────────────────┤
│     參數: char* s                   │ ← ESP + 4
├─────────────────────────────────────┤  ← ESP (堆疊指標)
│  int max_len (4 bytes)              │ ← 0xFFF8
├─────────────────────────────────────┤
│  int max_freq (4 bytes)             │ ← 0xFFF4
├─────────────────────────────────────┤
│  int left (4 bytes)                 │ ← 0xFFF0
├─────────────────────────────────────┤
│  int right (4 bytes)                │ ← 0xFFEC  [迴圈內部宣告]
├─────────────────────────────────────┤
│  freq[26] 陣列                      │ ← 0xF000 ~ 0xF067
│  (26 × 4 bytes = 104 bytes)         │
├─────────────────────────────────────┤
│                                     │
│ (堆疊向下增長 ↓)                     │
└─────────────────────────────────────┘

堆疊使用量統計:
  - freq[26]      : 104 bytes (26 個 int)
  - left          : 4 bytes
  - right         : 4 bytes (在迴圈中)
  - max_freq      : 4 bytes
  - max_len       : 4 bytes
  - 參數 (s, k)    : 8 bytes
  - 返回位址       : 8 bytes (64-bit 架構)
  ────────────────────────────
  總計            : ~144 bytes (加上對齊)
```

## 3. 搜尋過程追蹤

### 範例 1: s = "ABAB", k = 2

**初始狀態:**
```
字串: A B A B
位置: 0 1 2 3

變數初始化:
  left = 0
  right = (迴圈開始為 0)
  max_freq = 0
  max_len = 0
  freq[26] = [0, 0, 0, ...] 全為 0
```

---

#### 步驟 1: right = 0, s[0] = 'A'

```
字串狀態:
┌──┬──┬──┬──┐
│A │B │A │B │
├──┼──┼──┼──┤
│0 │1 │2 │3 │
└──┴──┴──┴──┘
 ▲ ◄─ 處理位置
 left=0

操作:
  1. freq['A' - 'A']++ → freq[0]++  [0 → 1]
  2. freq[0] = 1 > max_freq = 0 → max_freq = 1
  3. window_len = 0 - 0 + 1 = 1
  4. need_change = 1 - 1 = 0
  5. need_change (0) ≤ k (2) → 不需要縮小視窗
  6. current_len = 0 - 0 + 1 = 1
  7. 1 > max_len = 0 → max_len = 1

狀態 (步驟 1 後):
  freq[0] = 1 (A)
  freq[1] = 0 (B)
  left = 0, right = 0
  max_freq = 1, max_len = 1
```

---

#### 步驟 2: right = 1, s[1] = 'B'

```
字串狀態:
┌──┬──┬──┬──┐
│A │B │A │B │
├──┼──┼──┼──┤
│0 │1 │2 │3 │
└──┴──┴──┴──┘
 ▲  ▲ ◄─ 處理位置
 L  R

操作:
  1. freq['B' - 'A']++ → freq[1]++  [0 → 1]
  2. freq[1] = 1 不大於 max_freq = 1 (相等) → max_freq 保持 1
  3. window_len = 1 - 0 + 1 = 2
  4. need_change = 2 - 1 = 1

  6. current_len = 1 - 0 + 1 = 2
  7. 2 > max_len = 1 → max_len = 2

狀態 (步驟 2 後):
  freq[0] = 1 (A), freq[1] = 1 (B)
  left = 0, right = 1
  max_freq = 1, max_len = 2
```

## 5. 記憶體存取模式分析

### freq[26] 陣列的快取行為

```
快取友善性分析:

┌──────────────────────────────────────────────────┐
│            CPU L1 快取行 (64 bytes)              │
├──────────────────────────────────────────────────┤
│ freq[0-15] (4×16 = 64 bytes) 完全適配單個快取行 │
├──────────────────────────────────────────────────┤
└──────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────┐
│            CPU L1 快取行 (64 bytes)              │
├──────────────────────────────────────────────────┤
│ freq[16-25] 佔用 40 bytes (加上填充約 64 bytes)  │
├──────────────────────────────────────────────────┤
└──────────────────────────────────────────────────┘

快取命中率分析:
  - freq[26] = 104 bytes ≈ 1.6 個快取行
  - 字母頻率陣列幾乎適配兩個快取行 (256 位元 ≈ 2 × 128 位元)
  - 時間區域性 (temporal locality): 每個 right 迴圈都會存取 freq 一次
  - 空間區域性 (spatial locality): 高! 因為 s[i] 映射到固定的 freq[idx]

時間複雜度對快取的影響:
  - 迴圈執行 n 次，其中 n = strlen(s)
  - 每次迴圈:
    * freq[idx]++ 讀寫 (快取熱)
    * 比較運算無記憶體存取
    * left 指標縮小時額外 freq[s[left]]-- (仍在快取中)
  - 字串 s 的順序掃描:
    * 字串字元存儲在記憶體中,逐個讀取 (cache-friendly sequential read)
    * CPU 預取機制自動加載下一個快取行
```

### 與雜湊表 (Hash Map) 的比較

```
方案 1: freq[26] 陣列 (當前解法)
┌─────────────────────────────────┐
│ 優點:                           │
│ - 固定 O(1) 存取時間           │
│ - L1 快取適配 (104 bytes)      │
│ - 無雜湊碰撞                    │
│ - 記憶體連續分配               │
│ - CPU 預取友善                 │
│ - 分支預測友善                 │
└─────────────────────────────────┘

方案 2: 雜湊表 (Hash Map)
┌─────────────────────────────────┐
│ 缺點:                           │
│ - 動態分配 (不連續記憶體)       │
│ - 雜湊碰撞開銷                  │
│ - 指標追蹤 (多級記憶體存取)     │
│ - L1 快取未必適配               │
│ - 分支預測困難                  │
│ - 快取行汙染                    │
└─────────────────────────────────┘

快取行汙染對比:
┌──────────────────────────────────┐
│ freq[26] (數組):                 │
│  ┌───────────────────────────┐   │
│  │ freq[idx] ← CPU 快取命中  │   │
│  │ (下一個字符也在快取內)    │   │
│  └───────────────────────────┘   │
│                                  │
│ Hash Map:                        │
│  ┌────────────────────────────┐  │
│  │ hash(ch) → bucket idx      │  │
│  │ ↓ (L1 快取未必適配)        │  │
│  │ bucket → hash table 槽      │  │
│  │ ↓ (指標追蹤 L2/L3)         │  │
│  │ node → 鍵值對結構          │  │
│  │ (可能跨越多個快取行)       │  │
│  └────────────────────────────┘  │
└──────────────────────────────────┘

記憶體存取計數 (每個字符):
  - freq[26]: 1 × 記憶體讀寫 (快取熱)
  - Hash Map: 3~5 × 記憶體操作 (快取冷)
```

## 7. 關鍵演算法洞察

### 滑動視窗演算法的核心

```
■ 二指標維持不變式 (Invariant):
  視窗 [left, right] 內所有有效長度 ≤ 當前最優值
  (因此不需要回溯)

■ max_freq 的動態更新:
  max_freq 記錄視窗內最高頻率的字符
  這是貪心策略的關鍵:
    - 視窗長度 = (right - left + 1)
    - 需替換次數 = window_len - max_freq
    - 若 need_change ≤ k，視窗可擴展

■ 視窗縮小的時機:
  只在 need_change > k 時縮小
  (否則保持左指標不動,讓右指標繼續擴展)

■ max_freq 為何不會遞減?
  max_freq 記錄的是整個執行過程中見過的最高頻率
  即使視窗縮小,我們不會降低 max_freq
  原因: 我們只關心「最長」的有效視窗,
        舊的高頻率記錄仍然有效
```

### 記憶體走訪模式最優性

```
┌─────────────────────────────────────────────────┐
│ 記憶體走訪序列 (最優級別):                      │
├─────────────────────────────────────────────────┤
│                                                 │
│ 主迴圈中每次迭代:                              │
│  1. 讀 s[right]         (字串順序讀, 快取預取) │
```
