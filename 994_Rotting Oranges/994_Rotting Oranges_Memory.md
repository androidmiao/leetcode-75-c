# LeetCode 994 Rotting Oranges - 記憶體分析

## 1. 佇列記憶體佈局

多源 BFS 使用扁平化陣列來實現佇列，儲存 (row, col) 座標對。

### 佇列結構
```
int* queue           ── 指向 int 陣列的指標
int front = 0        ── 佇列前端指標
int rear = 0         ── 佇列後端指標
最大容量 = m*n*2 個 int 元素（每個座標佔 2 個 int）
```

### 記憶體佈局圖（基底位址 0xA000）

```
┌─────────────────────────────────────────────────────────────┐
│ queue 陣列 (整數扁平化儲存)                                  │
├─────────────────────────────────────────────────────────────┤
│ 偏移 │ 位址    │ 內容        │ 說明                          │
├─────┼─────────┼─────────────┼──────────────────────────────┤
│ 0   │ 0xA000  │ row[0]      │ 第 1 組座標：列號            │
│ 1   │ 0xA004  │ col[0]      │ 第 1 組座標：行號            │
│ 2   │ 0xA008  │ row[1]      │ 第 2 組座標：列號            │
│ 3   │ 0xA00C  │ col[1]      │ 第 2 組座標：行號            │
│ 4   │ 0xA010  │ row[2]      │ 第 3 組座標：列號            │
│ 5   │ 0xA014  │ col[2]      │ 第 3 組座標：行號            │
│ ... │ ...     │ ...         │ ...                          │
└─────┴─────────┴─────────────┴──────────────────────────────┘
```

### 佇列狀態視覺化

```
初始狀態（等待入隊）：
┌────────────────────────────────────────┐
│ 0 │ 0 │ ? │ ? │ ? │ ? │ ? │ ? │ ... │ ?│
└────────────────────────────────────────┘
 ▲front
 ▲rear

入隊 (0, 0)：
┌────────────────────────────────────────┐
│ 0 │ 0 │ ? │ ? │ ? │ ? │ ? │ ? │ ... │ ?│
└────────────────────────────────────────┘
 ▲front                  ▲rear

出隊 (0, 0)、入隊 (0, 1) 和 (1, 0)：
┌────────────────────────────────────────┐
│ 0 │ 0 │ 0 │ 1 │ 1 │ 0 │ ? │ ? │ ... │ ?│
└────────────────────────────────────────┘
       ▲front            ▲rear
```

### 指標追蹤
```
front = index / 2      ── 目前處理的座標組數
rear = index / 2       ── 下一個可用的座標組數
dequeue: row = queue[front*2], col = queue[front*2+1]
enqueue: queue[rear*2] = row, queue[rear*2+1] = col
```

## 2. 方向陣列記憶體配置

用來探索四鄰域（上、下、左、右）的相對位移。

### 記憶體佈局（基底位址 0xB000）

```
┌─────────────────────────────────────────────────────────┐
│ directions[8] 陣列 - 每個 int 佔 4 位元組                │
├─────┬─────────┬─────────┬──────────────────────────────┤
│ i   │ 位址    │ 值      │ 含義                          │
├─────┼─────────┼─────────┼──────────────────────────────┤
│ 0   │ 0xB000  │ -1      │ 上方：列號 -1                │
│ 1   │ 0xB004  │  0      │ (搭配行號  0)                │
├─────┼─────────┼─────────┼──────────────────────────────┤
│ 2   │ 0xB008  │  1      │ 下方：列號 +1                │
│ 3   │ 0xB00C  │  0      │ (搭配行號  0)                │
├─────┼─────────┼─────────┼──────────────────────────────┤
│ 4   │ 0xB010  │  0      │ 左方：列號  0                │
│ 5   │ 0xB014  │ -1      │ (搭配行號 -1)                │
├─────┼─────────┼─────────┼──────────────────────────────┤
│ 6   │ 0xB018  │  0      │ 右方：列號  0                │
│ 7   │ 0xB01C  │  1      │ (搭配行號 +1)                │
└─────┴─────────┴─────────┴──────────────────────────────┘

new_row = row + directions[i]
new_col = col + directions[i+1]
```

### 方向順序
```
directions = {-1, 0,    1, 0,    0, -1,   0, 1}
              ↑上方    ↓下方    ←左方    →右方
```

## 3. Grid 二維陣列記憶體配置

`int** grid` 是指標陣列，每個元素指向一列。

### 三維度示例 (m=3, n=3)

```
邏輯上的 grid：
    col 0  col 1  col 2
row 0 [2]   [1]    [1]
row 1 [1]   [1]    [0]
row 2 [0]   [1]    [1]
```

### 實際記憶體配置

```
grid 陣列本身（行指標陣列）@ 0xC000：
┌──────────────┬──────────────┬──────────────┬──────────┐
│ grid[0]      │ grid[1]      │ grid[2]      │ unused   │
│ @ 0xC000     │ @ 0xC008     │ @ 0xC010     │          │
├──────────────┼──────────────┼──────────────┤          │
│ 0xD000───────▶│ 0xD100───────▶│ 0xD200───────▶│          │
└──────────────┴──────────────┴──────────────┴──────────┘
      ↓              ↓              ↓

row 0 @ 0xD000：
┌────────┬────────┬────────┐
│ [0]=2  │ [1]=1  │ [2]=1  │
├────────┼────────┼────────┤
│0xD000  │0xD004  │0xD008  │
└────────┴────────┴────────┘

row 1 @ 0xD100：
┌────────┬────────┬────────┐
│ [0]=1  │ [1]=1  │ [2]=0  │
├────────┼────────┼────────┤
│0xD100  │0xD104  │0xD108  │
└────────┴────────┴────────┘

row 2 @ 0xD200：
┌────────┬────────┬────────┐
│ [0]=0  │ [1]=1  │ [2]=1  │
├────────┼────────┼────────┤
│0xD200  │0xD204  │0xD208  │
└────────┴────────┴────────┘
```

### 存取模式
```
grid[row][col] = *(grid[row] + col)
              = *(0xC000 + row*8)[col*4]

例：grid[1][2] = *(0xC008)[0xD108]
              = *(grid[1] + 2*4)
              = value at 0xD108
              = 0
```

## 6. 指標變數追蹤總表

### 範例 1：[[2,1,1],[1,1,0],[0,1,1]] — 結果：2

| 終止 | — | (2,2) | 佇列空，BFS 結束 | — | 0 | 2 |

### 範例 2：[[2,1,1],[0,1,1],[1,0,1]] — 結果：−1

| Step | Queue Front Addr @ | Dequeued (r,c) | 操作 | Grid 變化 | fresh_count | minutes |

- 範例 1：所有新鮮橙子都被感染，返回 minutes = 2
- 範例 2：(2,0) 被 0 包圍，無法感染，返回 −1

## 7. 記憶體存取模式分析

### 7.1 扁平陣列佇列 vs 鏈表佇列

#### 扁平陣列佇列（本實現）

```
記憶體佈局：
0xA000 ─────────────────────────────────────────────┐
  │ 0 │ 0 │ 1 │ 0 │ 0 │ 1 │ 1 │ 1 │ 0 │ 2 │ ... │
  └───┴───┴───┴───┴───┴───┴───┴───┴───┴───┴─────┘
  ◀───────────────────────────────────────────────▶
         連續、緊密排列的記憶體區塊

優勢：
✓ 高 L1/L2 快取命中率：連續記憶體區塊
✓ 低配置開銷：一次分配，無額外指標
✓ 簡單的前後指標運算：front++, rear++
✓ CPU 預取友善：順序記憶體存取

劣勢：
✗ 固定大小容量：m*n*2 必須預先分配
✗ 無法動態調整大小
✗ 浪費空間：若佇列元素數遠小於最大容量
```

#### 鏈表佇列（替代方案）

```
記憶體佈局：
┌───────┬────────┐      ┌───────┬────────┐      ┌───────┬────────┐
│(0,0)  │ next ──┼─────▶│(1,0)  │ next ──┼─────▶│(0,1)  │ next→●│
├───────┼────────┤      ├───────┼────────┤      ├───────┼────────┤
│0xD000 │0xD008  │      │0xD100 │0xD108  │      │0xD200 │0xD208  │
└───────┴────────┘      └───────┴────────┘      └───────┴────────┘
 ▲head                                                        ▲tail

優勢：
✓ 動態大小：依需要分配節點
✓ 記憶體效率：精確分配所需空間
✓ 彈性操作：無大小限制

劣勢：
✗ 低快取命中率：節點分散在堆記憶體中
✗ 高配置開銷：每個節點的額外指標（8 位元組）
✗ 間接存取：指標追蹤的 CPU 開銷
✗ 虛擬記憶體頁面故障風險較高
```

### 7.2 行優先 Grid 遍歷與快取友善性

#### 行優先遍歷（row-major，本實現）

```
邏輯順序：
(0,0) → (0,1) → (0,2) → (1,0) → (1,1) → (1,2) → (2,0) → (2,1) → (2,2)

記憶體位址：
0xD000 → 0xD004 → 0xD008 │ 跳躍 │ 0xD100 → 0xD104 → 0xD108 │ ...
         ◀───連續────────▶

快取線（假設 64 位元組）：
┌─────────────────────────────────────────────────────────────┐
│ 0xD000(4B) │ 0xD004(4B) │ 0xD008(4B) │ ... │ 其他資料(48B) │
└─────────────────────────────────────────────────────────────┘
 ◀───────────────────────────────────────────────────────────▶
           單一快取線內的 3 個元素，高命中率

優勢：
✓ 快取侷部性：相同列的元素連續存儲
✓ 預取友善：CPU 預取器可預測下一個元素
✓ 頁面侷部性：列通常在同一虛擬記憶體頁面
```

#### 列優先遍歷（column-major，對比）

```
邏輯順序：
(0,0) → (1,0) → (2,0) → (0,1) → (1,1) → (2,1) → (0,2) → (1,2) → (2,2)

記憶體位址：
0xD000 │ 跳躍 │ 0xD100 │ 跳躍 │ 0xD200 │ 跳躍 │ 0xD004 │ ...

快取行為：
快取缺失發生在每次行跳躍時，導致頻繁的快取重新載入

劣勢：
✗ 快取缺失率高：不同行之間存在跳躍
✗ 虛擬記憶體頁面故障率提高
✗ CPU 預取難以優化
```

### 7.3 二級指標 `int** grid` 的快取影響

#### 實際記憶體佈局

```
邏輯上的二維陣列訪問：grid[i][j]

CPU 執行步驟：
1. 讀取 grid 指標本身 @ 0xC000       ◀─ 簽訂
2. 讀取 grid[i] 值（行指標）@ 0xC000 ◀─ 可能快取命中
3. 計算 grid[i] + j 位址
4. 讀取 grid[i][j] 值 @ 0xDxxx       ◀─ 可能快取缺失

真實記憶體路徑：
┌──────────────┐        ┌────────────────┐        ┌──────────────┐
│ grid 指標    │───────▶│ grid[i] 值     │───────▶│ grid[i][j]   │
│ @ 0xC000     │        │ @ 0xD000+...   │        │ @ 0xD100+... │
└──────────────┘        └────────────────┘        └──────────────┘
     (4 Byte)                (4 Byte)                   (4 Byte)
    1 快取線              1 快取線                  可能不同快取線
```

#### 快取侷部性分析

```
時間侷部性（Temporal）：
❌ 較差：grid[i] 指標讀取 1 次，之後長期未再訪問
✓ 改善：在內層迴圈中重複使用 grid[i]

空間侷部性（Spatial）：
❌ 較差：grid 指標陣列 @ 0xC000
         grid 行陣列 @ 0xD000, 0xD100, 0xD200（分散）
         最終資料值 @ 各行起始地址（快取線重疊可能低）

✓ 改善：預先計算行指標，減少間接訪問
        int* row = grid[i];
        for (int j = 0; j < n; j++) {
            value = row[j];  // 直接存取，快取友善
        }
```

### 7.4 最佳化策略

#### 策略 1：快取友善的佇列遍歷

```c
// 當前實現（已最佳化）
for (int i = front; i < rear; i++) {
    int row = queue[i*2];        // 連續讀取，快取友善
    int col = queue[i*2+1];
    // 處理 (row, col)
}
```

#### 策略 2：行快取優化

```c
// 予先快取行指標
int* current_row = grid[row];
for (int d = 0; d < 4; d++) {
    int new_row = row + directions[d*2];
    int new_col = col + directions[d*2+1];

    if (new_row >= 0 && new_row < m && new_col >= 0 && new_col < n) {
        int* nr = grid[new_row];     // 只在切換行時重新載入
        if (nr[new_col] == 1) {
            // 感染邏輯
        }
    }
}
```

#### 策略 3：動態佇列（連結清單替代）

```c
typedef struct Node {
    int row, col;
    struct Node* next;
} Node;

Node* head = NULL, *tail = NULL;

// 優勢：精確的記憶體使用，無浪費
// 劣勢：間接存取損失快取命中率

// 時間複雜度：O(m*n)（相同）
// 空間複雜度：O(m*n)（相同）
// 快取命中率：□ 降低 25-40%（取決於記憶體分配器）
```

### 7.5 記憶體效率總結

| 指標 | 扁平陣列佇列 | 鏈表佇列 | int** grid | 真 2D 陣列 |
|------|----------|---------|-----------|---------|
| **快取命中率** | ⭐⭐⭐⭐⭐ 極高 | ⭐⭐ 低 | ⭐⭐⭐ 中 | ⭐⭐⭐⭐⭐ 極高 |
| **分配開銷** | ⭐⭐⭐⭐⭐ 低 | ⭐ 高 | ⭐⭐⭐ 中 | ⭐⭐⭐ 中 |
| **記憶體浪費** | ⭐⭐ 中等 | ⭐⭐⭐⭐⭐ 無 | ⭐⭐⭐⭐ 少 | ⭐⭐⭐⭐ 少 |
| **間接訪問** | 無 | 多 | 有 | 無 |
| **預取友善** | ⭐⭐⭐⭐⭐ 優 | ⭐ 差 | ⭐⭐⭐ 中 | ⭐⭐⭐⭐⭐ 優 |
| **虛擬記憶體頁面故障** | 低 | 中 | 中 | 低 |

**結論**：本實現（扁平陣列 + int** grid）在實踐中快取效率良好，權衡了靈活性與效能。若追求極限效能，應使用 `int grid[m][n]` 真二維陣列，但需要在編譯時確定 m 和 n。
