## Linux 相關原始碼位置

| 檔案 | 角色 |
|---|---|
| `lib/find_bit.c` | 位操作搜尋函式，概念上與方法 6（位集合和）相關 |
| `include/linux/list.h` | 核心雙向鏈結串列實作，`list_for_each_entry()` 巨集體現了指標追蹤模式 |
| `net/core/neighbour.c` | ARP / NDP 鄰居表使用循環檢測來偵測路由迴圈 |
| `fs/namei.c` | 路徑名查找中的符號連結循環檢測（`ELOOP`），最多跟隨 40 層符號連結 |
| `kernel/locking/lockdep.c` | 鎖依賴圖的循環檢測——偵測潛在死鎖 |
| `drivers/net/ethernet/` (各驅動) | 許多網路驅動使用環形緩衝區 (ring buffer)，其 head/tail 指標追蹤本質上是循環結構中的快慢指標模式 |

### 最直接的對應：`fs/namei.c` 中的符號連結循環檢測

Linux 核心在路徑名解析時必須偵測符號連結循環。當 `/a -> /b` 且 `/b -> /a` 時，如果不限制就會無限遞迴。核心使用一個計數器（`nd->total_link_count`，上限 40）來檢測這種情況，概念上等同於 Floyd 演算法中「必定存在循環」的前提——映射空間有限，重複訪問不可避免。

### 更深層的對應：`lockdep` 死鎖檢測

`lockdep` 子系統維護一個鎖的有向圖。當新的鎖依賴被添加時，它會在圖上做 BFS/DFS 搜尋循環。如果找到循環，就表示存在潛在死鎖——這和 LeetCode 287 的核心洞察完全一致：**「在有限映射空間中，重複映射必定造成循環」**。

- LeetCode 287：n+1 個值映射到 [1,n] → 必有循環入口（重複數字）
- lockdep：n 個鎖互相等待 → 若形成環就是死鎖

## Kernel vs. LeetCode 實作對照

| 維度 | LeetCode 287（Floyd 龜兔） | Linux `lockdep`（鎖依賴循環檢測） | Linux `namei.c`（符號連結循環） |
|---|---|---|---|
| 資料結構 | 隱式鏈表（陣列索引跳轉） | 有向圖（鎖依賴關係） | 鏈式路徑（符號連結鏈） |
| 循環檢測方法 | Floyd 快慢指標 | BFS/DFS 圖遍歷 | 計數器上限（40 次） |
| 時間複雜度 | O(n) | O(V+E)，V=鎖數量，E=依賴數量 | O(40) = O(1)（硬上限） |
| 空間複雜度 | O(1) | O(V) 用於已訪問集合 | O(1)（僅一個計數器） |
| 找到循環入口？ | 是——第二階段精確定位 | 是——回溯路徑報告完整的死鎖鏈 | 否——僅報告 ELOOP 錯誤 |
| 能處理多重循環？ | 否——僅找到一個入口 | 是——每次新增依賴都檢查 | 不適用 |

## 主管 Code Review 角度考題

### Q1: 為什麼選擇 Floyd 而不是直接用 visited 陣列標記？

**期望回答方向：** Floyd 的空間複雜度為 O(1)，而 visited 陣列需要 O(n)。在嵌入式系統或記憶體受限的環境中，這個差異是關鍵的。此外，Floyd 不修改原始資料，這在多執行緒環境中是重要的不可變性保證。

**後續探針：** 如果陣列值的範圍不是 [1, n] 而是任意的，Floyd 還適用嗎？（答：不適用——隱式鏈表的前提會被打破。）

**真正測試的能力：** 演算法選擇的 trade-off 分析、記憶體約束意識。

### Q2: 核心中的 `lockdep` 為什麼不用 Floyd 演算法做循環檢測？

**期望回答方向：** Floyd 演算法只能在「函數式映射」（每個節點恰好一個後繼）的結構上運作。鎖依賴圖是一般有向圖，每個節點可能有多個後繼，所以需要 BFS/DFS。Floyd 的快慢指標假設路徑是唯一確定的，在分支圖上沒有意義。

**後續探針：** 那什麼時候圖會退化成函數式映射？（答：每個節點出度恰好為 1 時，例如陣列索引跳轉、鏈表 next 指標。）

**真正測試的能力：** 圖論基礎、演算法適用條件的精確理解。

### Q3: 如果 `nums` 陣列非常大（例如 10^8 個元素），Floyd 的快取行為如何？

**期望回答方向：** Floyd 的存取模式是跳躍式的（根據陣列值決定下一個訪問位置），這對 CPU 快取非常不友好——每次跳轉幾乎都是 cache miss。相比之下，排序方法（方法 1）雖然時間複雜度更高（O(n log n)），但排序本身和後續的線性掃描都有良好的空間局部性。在實際硬體上，對於特定大小的輸入，O(n log n) 的排序可能比 O(n) 的 Floyd 更快。

**後續探針：** 能否改進 Floyd 的快取行為？（答：本質上很難，因為跳轉目標由資料決定。可以考慮先做一次線性掃描建立某種快取提示，但那就不是 O(1) 空間了。）

**真正測試的能力：** 快取局部性意識、理論複雜度與實際性能的差距理解。

### Q4: Phase 2 中，為什麼龜從相遇點出發、兔從起點出發就能找到循環入口？請用程式碼層面解釋，不要只背公式。

**期望回答方向：** 相遇時龜走了 d 步，兔走了 2d 步。兔多走的 d 步恰好是循環長度的整數倍（因為兔在循環內多繞了若干圈才追上龜）。所以 d = kC。龜在相遇點距離循環入口 `d - F = kC - F` 步（沿循環方向）。從起點走 F 步到達入口，從相遇點走 `kC - (d-F)` 步也到達入口。由於 `d = kC`，從相遇點走 F 步就會到達入口。所以兩者同步走 F 步就同時到達入口。

**真正測試的能力：** 數學推理能力、能否用直覺而非死記硬背來解釋。

### Q5: 在核心的環形緩衝區（ring buffer）實作中，head 和 tail 指標的追蹤和 Floyd 有什麼異同？

**期望回答方向：** 環形緩衝區的 head/tail 是生產者/消費者指標，它們以不同速度前進但**不會相遇**（head == tail 表示空或滿，是邊界條件，不是循環檢測）。Floyd 的快慢指標則是刻意讓它們相遇以證明循環的存在。兩者都在循環結構中操作，但目的相反：ring buffer 要避免追上，Floyd 要追上。

**真正測試的能力：** 對循環資料結構的深度理解、區分相似模式的能力。

## 面試加分總結

- **展示跨領域知識：** 提到 `lockdep` 的死鎖檢測和 `namei.c` 的符號連結循環檢測，說明你理解循環檢測在系統軟體中的廣泛應用，而非僅停留在演算法題層面。
- **快取局部性討論：** 主動提出 Floyd 跳躍存取的 cache 問題，展示你對硬體特性的敏感度——這在 Google 等公司的 L5+ 面試中是加分項。
- **數學證明能力：** 能用自己的話推導 Phase 2 的正確性，而非背誦公式，展現真正的理解。
- **鴿籠原理的連結：** 從「為什麼循環必定存在」出發，連結到鴿籠原理，再延伸到核心中的類似設計模式（有限資源→必有衝突），形成完整的認知框架。
- **ring buffer 對比：** 提出 Floyd 快慢指標與環形緩衝區 head/tail 的異同分析，展示你能在相似模式間做出精確區分。
