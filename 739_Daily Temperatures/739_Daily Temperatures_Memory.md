# LeetCode 739 每日溫度 - 記憶體位址與陣列索引圖解

## 1. 記憶體佈局 (Memory Layout)

三個陣列的基礎記憶體配置 (假設固定基礎位址，每個 int = 4 bytes):

```
┌─────────────────────────────────────────────────────────────────────┐
│                      記憶體佈局示意圖                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  temperatures 陣列                                                   │
│  ├─ 基礎位址: 0xA000                                                │
│  ├─ 大小: n * 4 bytes                                              │
│  └─ 存放: 輸入溫度值                                                │
│                                                                      │
│  result 陣列 (calloc'd)                                             │
│  ├─ 基礎位址: 0xB000                                                │
│  ├─ 大小: n * 4 bytes                                              │
│  ├─ 初始化: 全 0                                                    │
│  └─ 存放: 輸出 (每個位置到更高溫度的天數)                          │
│                                                                      │
│  stack 陣列 (malloc'd)                                              │
│  ├─ 基礎位址: 0xC000                                                │
│  ├─ 大小: n * 4 bytes                                              │
│  ├─ 存放: 索引 (單調遞減堆疊)                                       │
│  └─ top 指標: 目前堆疊頂端位置 (-1 = 空)                            │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

## 3. 陣列索引版對應

堆疊中的每個元素都是一個索引，指向 temperatures 陣列。當從堆疊彈出時，索引用來：
1. 查閱 temperatures 值: `temperatures[stack[top]]`
2. 寫入 result 值: `result[stack[top]] = i - stack[top]`

```
示例: 當 i=1, temperatures[1]=74 > temperatures[0]=73
     堆疊中的 stack[0]=0 被彈出

     索引對應關係:
     ┌─────────────────────┐
     │ stack[0] = 0        │ ───▶ temperatures[0] = 73 (目標溫度值)
     │                     │
     │ 計算結果:           │
     │ result[0] = 1-0 = 1 │ ───▶ result[0] = 1 (1 天後更高)
     └─────────────────────┘

陣列間的資料流:
temperatures[i] ──▶ (比較) ◀── temperatures[stack[top]]
                        │
                        ▼
                   計算天數: i - stack[top]
                        │
                        ▼
                    result[stack[top]] = days
```

## 6. 記憶體存取模式分析 (Memory Access Pattern Analysis)

### 6.1 三陣列的存取模式

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    存取模式與快取行為分析                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                         │
│ temperatures[] 陣列:                                                     │
│  ├─ 存取模式: 線性掃描 (i=0 → n-1)                                      │
│  │  temperatures[i] 在每個步驟一次性存取                                │
│  │  temperatures[stack[top]] 不規則存取                                │
│  │                                                                     │
│  ├─ 存取順序:                                                          │
│  │  Step 0: temp[0]                                                   │
│  │  Step 1: temp[1], temp[0] (堆疊)  ◄── 亂序                         │
│  │  Step 2: temp[2], temp[1] (堆疊)                                   │
│  │  Step 3: temp[3], temp[2] (堆疊)                                   │
│  │  ...                                                               │
│  │                                                                     │
│  ├─ 快取行為:                                                          │
│  │  ✓ 主線性掃描: 優秀的快取局部性 (sequential)                       │
│  │  ✗ 堆疊查找: 差的快取局部性 (random access)                         │
│  │  ├─ 快取擊中率: 取決於堆疊中最多有多少個元素                        │
│  │  └─ 最壞情況: n 個元素全在堆疊中，每個都需要讀 temp 陣列           │
│  │                                                                     │
│  └─ 存取大小: O(n + m)                                                 │
│     其中 m = 所有 while 迴圈彈出的總次數 (≤ n)                        │
│     實際: O(2n) = O(n) 線性                                            │
│                                                                         │
│ result[] 陣列:                                                          │
│  ├─ 存取模式: 非線性寫入                                               │
│  │  result[prev_index] = i - prev_index  (prev_index 不規則)           │
│  │                                                                     │
│  ├─ 存取順序 (Example 1):                                              │
│  │  Step 1: 寫 result[0]                                              │
│  │  Step 2: 寫 result[1]                                              │
│  │  Step 3: 無                                                        │
│  │  Step 4: 無                                                        │
│  │  Step 5: 寫 result[4], 寫 result[3]  ◄── 反序                      │
│  │  Step 6: 寫 result[5], 寫 result[2]                                │
│  │  Step 7: 無                                                        │
│  │                                                                     │
│  ├─ 快取行為:                                                          │
│  │  ✗ 寫模式: 差的快取局部性 (隨機 writes)                             │
│  │  ├─ 快取寫入衝突: 多個不同位置的寫入                               │
│  │  └─ 寫回成本: 每次寫都可能觸發新快取行                             │
│  │                                                                     │
│  └─ 存取大小: O(n)  (每個元素最終寫入一次)                            │
│     平均: 所有元素都被寫入 (初始化為 0，然後更新或保留)                │
│                                                                         │
│ stack[] 陣列:                                                           │
│  ├─ 存取模式: LIFO (堆疊操作)                                          │
│  │  push: stack[++top] = i                                            │
│  │  pop:  prev_index = stack[top--]                                   │
│  │                                                                     │
│  ├─ 存取順序 (Example 1):                                              │
│  │  Step 0: 寫 stack[0]      (top 從 -1 → 0)                         │
│  │  Step 1: 讀 stack[0]，寫 stack[0]  (top 從 0 → 0)                 │
│  │  Step 2: 讀 stack[0]，寫 stack[0]  (top 從 0 → 0)                 │
│  │  Step 3: 寫 stack[1]      (top 從 0 → 1)                         │
│  │  Step 4: 寫 stack[2]      (top 從 1 → 2)                         │
│  │  Step 5: 讀 stack[2],stack[1]，寫 stack[1]  (top 從 2 → 1)       │
│  │  ...                                                               │
│  │                                                                     │
│  ├─ 快取行為:                                                          │
│  │  ✓ 優秀的快取局部性 (LIFO 棧操作)                                  │
│  │  ├─ 堆疊頂端 (top) 總是在快取中                                   │
│  │  └─ 相鄰的堆疊條目共享快取行                                       │
│  │  ├─ 快取擊中率: 接近 100% (堆疊小於快取行大小)                    │
│  │  └─ 存取延遲: 1 個時鐘週期 (如果常駐)                             │
│  │                                                                     │
│  └─ 存取大小: O(n)  (n 次推 + 最多 n 次彈)                           │
│     實際: 2n 個操作，但局部性優秀                                      │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

### 6.2 快取效能比較: 單調棧 vs 暴力法

```
┌────────────────────────────────────────────────────────────────────────┐
│             演算法        │  時間複雜度  │  快取局部性  │  總讀寫    │
├─────────────────────────────────────────────────────────────────────────┤
│ 單調棧 (Monotonic Stack)  │  O(n)       │  優秀        │  ~3n     │
│ ├─ temperatures[] 線性掃: O(n)                                        │
│ ├─ temperatures[] 堆疊查: O(m), m≤n  ◄── 局部性差但總量小             │
│ ├─ result[] 寫:          O(n)                                        │
│ ├─ stack[] 操作:         O(n)         ✓ 快取友善                    │
│ └─ 總計: O(n) 時間 + O(n) 空間                                       │
│                                                                      │
│ 暴力法 (Brute Force)      │  O(n²)      │  差          │  ~n²     │
│ ├─ 外迴圈 i:             O(n)                                       │
│ ├─ 內迴圈 j>i:           O(n)                                       │
│ │  ├─ 每次比較: temperatures[j] > temperatures[i]                 │
│ │  └─ 快取缺失: 高機率 (隨機 j 存取)                               │
│ ├─ result[] 寫:          O(n)                                       │
│ └─ 總計: O(n²) 時間 + O(n) 空間                                     │
│                                                                      │
│ 性能差異 (n=10,000):                                                │
│ ├─ 單調棧:               ~30,000 操作  + 1-2 次快取缺失/操作         │
│ ├─ 暴力法:               ~100,000,000 操作 + 10-20 次快取缺失       │
│ ├─ 速度比: 暴力法 ≈ 10,000倍慢                                      │
│ └─ 快取缺失比: 暴力法 ≈ 50-100倍多                                  │
│                                                                      │
└────────────────────────────────────────────────────────────────────────┘
```

## 關鍵洞察 (Key Insights)

1. **單調棧的威力**: 通過維持遞減順序，我們確保：
   - 每個元素最多被推一次和彈一次
   - 總時間複雜度 O(n)，不是 O(n²)

2. **記憶體佈局**: 三個陣列在不同位址 (0xA000, 0xB000, 0xC000)：
   - temperatures: 輸入，線性存取
   - result: 輸出，隨機寫入
   - stack: 工作空間，LIFO 存取 (快取友善)

3. **最壞情況**: 單調遞減輸入使堆疊達到最大深度 n
   - 但這時 while 迴圈執行 0 次
   - 結果全是 0，無計算開銷

4. **快取效能**:
   - 堆疊操作: 優秀 (L1 快取常駐)
   - 線性掃描: 優秀 (順序讀)
   - 隨機寫: 差 (但總量有限)

5. **對比暴力法**: O(n²) 時間 + 巨大快取缺失
   - n=1000: 單調棧 ~2000 操作 vs 暴力法 ~1,000,000 操作
   - 實際速度差異: 100-500 倍
