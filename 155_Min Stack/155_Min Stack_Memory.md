# 155. Min Stack — 記憶體位址 / 陣列索引圖解

> 以下所有記憶體位址皆為假設值，用於說明指標與陣列的關係。

---

## 1. Struct 記憶體佈局

```c
typedef struct {
    int* data;       // 8 bytes (64-bit pointer)
    int* min_stack;  // 8 bytes (64-bit pointer)
    int top;         // 4 bytes
    int capacity;    // 4 bytes
} MinStack;          // Total: 24 bytes (no padding needed)
```

## 2. 完整資料結構記憶體配置

### push(-2), push(0), push(-3) 後的完整狀態

**位址對應表：**

| 物件 | 假設位址 | 大小 |
|------|----------|------|
| `MinStack obj` | 0xA000 | 24 bytes |
| `data[]` | 0xC000 | 4000 bytes (1000 × 4) |
| `min_stack[]` | 0xD000 | 4000 bytes (1000 × 4) |

|--------|------------|-----------|-----------------|----------------|

| 1 | 0xC004 | 0 | 0xD004 | -2 |

## 3. 陣列索引版對應

Min Stack 不涉及樹結構，因此沒有 `2*i+1` 類型的索引公式。但兩個平行陣列的索引完全同步：

```
index:        0       1       2
            ┌───┐   ┌───┐   ┌───┐
data[]:     │-2 │   │ 0 │   │-3 │
            └───┘   └───┘   └───┘
              ↕       ↕       ↕      ← 相同索引，一對一對應
            ┌───┐   ┌───┐   ┌───┐
min_stack[]:│-2 │   │-2 │   │-3 │
            └───┘   └───┘   └───┘
```

**索引關係：**
- `min_stack[0] = data[0]`（第一個元素自己是最小值）
- `min_stack[i] = min(data[i], min_stack[i-1])`（遞推公式）

## 4. 操作過程追蹤

### Example 1 追蹤

#### 操作 1: `push(-2)`

```
top: -1 → 0

data[0] = -2        min_stack[0] = -2
   ┌────┐              ┌────┐
   │ -2 │              │ -2 │
   └────┘              └────┘
     ↑ top=0              ↑ top=0

因為 top==0（棧為空），min_stack[0] = val = -2
```

| 步驟 | 位址 | 值 | 比較 | 動作 |
|------|------|-----|------|------|
| 1 | 0xC000 | -2 | top==0（空棧） | `data[0]=-2`, `min_stack[0]=-2` |

**指標移動：** `top: -1 → 0`

---

#### 操作 2: `push(0)`

```
top: 0 → 1

data[]:        min_stack[]:
┌────┬────┐    ┌────┬────┐
│ -2 │  0 │    │ -2 │ -2 │
└────┴────┘    └────┴────┘
  [0]  [1]       [0]  [1]
        ↑ top=1        ↑ top=1

min(0, min_stack[0]) = min(0, -2) = -2
所以 min_stack[1] = -2
```

| 步驟 | 位址 | 值 | 比較 | 動作 |
|------|------|-----|------|------|
| 1 | 0xD000 | -2 | `curr_min = min_stack[0] = -2` | 讀取前一層最小值 |
| 2 | 0xC004 | 0 | `0 < -2`? → NO | `min_stack[1] = -2`（保留舊最小值） |

**指標移動：** `top: 0 → 1`

---

#### 操作 3: `push(-3)`

```
top: 1 → 2

data[]:              min_stack[]:
┌────┬────┬────┐     ┌────┬────┬────┐
│ -2 │  0 │ -3 │     │ -2 │ -2 │ -3 │
└────┴────┴────┘     └────┴────┴────┘
  [0]  [1]  [2]        [0]  [1]  [2]
              ↑ top=2              ↑ top=2

min(-3, min_stack[1]) = min(-3, -2) = -3
所以 min_stack[2] = -3   ← 新的全域最小值！
```

| 步驟 | 位址 | 值 | 比較 | 動作 |
|------|------|-----|------|------|
| 1 | 0xD004 | -2 | `curr_min = min_stack[1] = -2` | 讀取前一層最小值 |
| 2 | 0xC008 | -3 | `-3 < -2`? → YES | `min_stack[2] = -3`（更新為新最小值） |

**指標移動：** `top: 1 → 2`

---

#### 操作 4: `getMin()`

```
直接回傳 min_stack[top] = min_stack[2] = -3

不需要任何遍歷或比較 — O(1)！
```

---

#### 操作 5: `pop()`

```
top: 2 → 1

data[]:              min_stack[]:
┌────┬────┬────┐     ┌────┬────┬────┐
│ -2 │  0 │(-3)│     │ -2 │ -2 │(-3)│     ← [2] 邏輯上已移除
└────┴────┴────┘     └────┴────┴────┘
  [0]  [1]             [0]  [1]
        ↑ top=1              ↑ top=1

只需 top--，不需要清除 data[2] 或 min_stack[2]
下次 push 會覆蓋它們
```

**指標移動：** `top: 2 → 1`

---

#### 操作 6: `top()`

```
回傳 data[top] = data[1] = 0
```

---

#### 操作 7: `getMin()`

```
回傳 min_stack[top] = min_stack[1] = -2

pop() 之後，top 回到 1，min_stack[1] = -2 已經是正確的最小值
不需要重新計算！這就是預計算的威力。
```

## 5. 指標變數追蹤總表

### Example 1 完整追蹤

| 步驟 | 操作 | top (舊→新) | 存取位址 | 值 | 比較 | 動作 |
|------|------|-------------|----------|-----|------|------|
| 1 | push(-2) | -1→0 | 0xC000, 0xD000 | -2 | top==0 | `data[0]=-2`, `min_stack[0]=-2` |
| 2 | push(0) | 0→1 | 0xC004, 0xD004 | 0 | `0 < -2`? NO | `data[1]=0`, `min_stack[1]=-2` |
| 3 | push(-3) | 1→2 | 0xC008, 0xD008 | -3 | `-3 < -2`? YES | `data[2]=-3`, `min_stack[2]=-3` |
| 4 | getMin() | 2 | 0xD008 | -3 | — | 回傳 -3 |
| 5 | pop() | 2→1 | — | — | — | `top--` |
| 6 | top() | 1 | 0xC004 | 0 | — | 回傳 0 |
| 7 | getMin() | 1 | 0xD004 | -2 | — | 回傳 -2 |

**統計：**
- 總操作次數：7
- 陣列寫入次數：6（3 × push，每次寫入 data 和 min_stack 各一次）
- 陣列讀取次數：5（2 × getMin + 1 × top + 2 × 讀取 `min_stack[top-1]` 做比較）
- 最大棧深度：3

## 6. 記憶體存取模式比較

| 面向 | 平行陣列方式（我們的實作） | 鏈結串列 + 配對方式 |
|------|--------------------------|---------------------|
| **push 存取** | `data[top]` + `min_stack[top]` — 兩次循序寫入 | `malloc` 新節點 + 寫入 value/min 欄位 |
| **快取行為** | 優良 — `data[]` 和 `min_stack[]` 各自連續，循序存取 cache-friendly | 差 — 每個節點獨立 `malloc`，位址不連續 |
| **空間額外開銷** | 零 — 只有兩個 `int[]` | 每個節點 16-24 bytes 的 `malloc` overhead + 指標欄位 |
| **pop 存取** | `top--` 一條指令 | `free` 節點 — 涉及記憶體回收 |
| **getMin** | 一次陣列讀取 | 一次指標解引用 |

### 為什麼平行陣列更好？

1. **快取局部性（Cache Locality）：** `data[0], data[1], data[2], ...` 在記憶體中連續，CPU 讀取一個 cache line（通常 64 bytes = 16 個 int）就能涵蓋 16 個棧元素。鏈結串列的節點散落在 heap 中，每次 `->next` 都可能 cache miss。

2. **避免 malloc/free 開銷：** 鏈結串列每次 push 需要 `malloc`，每次 pop 需要 `free`。我們的實作只在 `realloc` 時有配置成本，而 `realloc` 極少發生（容量翻倍策略，amortized O(1)）。

3. **分支預測友好：** 陣列存取的模式固定（每次 `top++` 然後寫入），CPU 分支預測器容易預測。

## 7. 堆疊記憶體對比（迭代 vs. 遞迴）

Min Stack 的所有操作都是迭代的，不使用遞迴，因此不會消耗 call stack：

```
迭代版（我們的實作）— call stack 使用量：

    ┌──────────────────┐
    │ minStackPush()   │  ← 固定 stack frame，約 32 bytes
    ├──────────────────┤
    │ caller (main)    │
    └──────────────────┘

    無論 push 多少元素，call stack 深度永遠是 O(1)
```

假設有人用遞迴方式實作 getMin（掃描整個棧找最小值）：

```
遞迴版 getMin — call stack 使用量（n=5 時）：

    ┌──────────────────┐
    │ getMin(depth=5)  │  ← 每層約 48 bytes
    ├──────────────────┤
    │ getMin(depth=4)  │
    ├──────────────────┤
    │ getMin(depth=3)  │
    ├──────────────────┤
    │ getMin(depth=2)  │
    ├──────────────────┤
    │ getMin(depth=1)  │
    ├──────────────────┤
    │ caller (main)    │
    └──────────────────┘

    call stack 深度 O(n)，可能 stack overflow！
    （Linux 預設 thread stack 8MB，每層 48 bytes → 約 170,000 層才溢出，
      但 kernel stack 只有 8-16KB → 約 170-340 層就溢出！）
```

**結論：** 使用輔助陣列預計算最小值，不僅讓 getMin 從 O(n) 降到 O(1)，還完全避免了遞迴帶來的棧深度風險。

## 8. 退化情況分析

### 最壞情況：嚴格遞減序列

輸入 `push(5), push(4), push(3), push(2), push(1)`：

```
data[]:              min_stack[]:
┌───┬───┬───┬───┬───┐  ┌───┬───┬───┬───┬───┐
│ 5 │ 4 │ 3 │ 2 │ 1 │  │ 5 │ 4 │ 3 │ 2 │ 1 │
└───┴───┴───┴───┴───┘  └───┴───┴───┴───┴───┘
```

`min_stack[]` 的每個值都不同 — 在方法二/三中，追蹤棧與主棧一樣大，無法節省空間。在我們的方法一中，`min_stack[]` 本來就和 `data[]` 一樣大，所以沒有差異。

### 最佳情況（方法二/三）：嚴格遞增序列

輸入 `push(1), push(2), push(3), push(4), push(5)`：

```
方法一（我們的實作）：
data[]:              min_stack[]:
┌───┬───┬───┬───┬───┐  ┌───┬───┬───┬───┬───┐
│ 1 │ 2 │ 3 │ 4 │ 5 │  │ 1 │ 1 │ 1 │ 1 │ 1 │  ← 全部是 1，冗餘！
└───┴───┴───┴───┴───┘  └───┴───┴───┴───┴───┘

方法二：
main_stack:  [1, 2, 3, 4, 5]
min_tracker: [1]              ← 只有一個元素！大幅節省空間！

方法三：
main_stack:  [1, 2, 3, 4, 5]
min_tracker: [(1, 1)]         ← 也只有一個配對
```

**結論：** 方法一在遞增序列時空間效率最差（`min_stack` 全部儲存相同值），但實作最簡單、cache 最友好。方法二/三在平均情況下空間更優，但增加了實作複雜度和 edge case 風險。
