# LeetCode 1. Two Sum — 記憶體分析

## 1. Struct 記憶體佈局

### HashEntry 結構定義

```c
typedef struct {
    int used;      // offset 0, 4 bytes (flag: 0 = 未使用, 1 = 已使用)
    int key;       // offset 4, 4 bytes (nums 元素值)
    int index;     // offset 8, 4 bytes (原陣列索引)
} HashEntry;       // 總計 12 bytes
```

### 單一入口記憶體圖示（假設地址 0xA000）

```

│  偏移量    │ 欄位名稱 │ 位元組範圍  │  值示例    │
```

- nums 大小: 4
- 雜湊表大小: 2 × 4 + 1 = 9
- 每個 HashEntry: 12 bytes

**記憶體配置假設：**
- 雜湊表起始地址: 0xB000
- nums 陣列起始地址: 0xC000

## 4. 搜尋／操作過程追蹤


## 6. 記憶體存取模式比較

### 開放定址法 (Open Addressing) — 本解法

**特點:**
```
記憶體配置:
┌───────────────────────────────────────────────┐
│ 連續的 HashEntry 陣列                          │
│ 0xB000: [entry0] [entry1] [entry2] ... [entry8] │
└───────────────────────────────────────────────┘
         ▲連續記憶體, 優良快取局部性
```

| 特性 | 優點 | 缺點 |
|------|------|------|
| **記憶體配置** | 連續陣列,快取友善 | 可能聚集 (clustering) |
| **快取效能** | 線性存取,高效預取 | 探測可能跳躍 |
| **空間效率** | 無指標開銷 | 需要更大的表 (負載因子低) |
| **搜尋時間** | 平均 O(1), 最差 O(n) | 聚集導致效能惡化 |
| **整數計算** | 簡單的取模運算 | 可能碰撞,需線性探測 |

**記憶體存取順序 (Example 1):**
```
步驟 1: 計算 hashIndex = 2
        存取 0xB018 (slot 2)
        ▼
        0xB000  0xB00C  0xB018 ◀─ 存取
                        │
        線性記憶體       │

步驟 2: 計算 hashIndex = 7 (應該查詢 complement)
        線性探測從 slot 2 開始...
        直接存取 0xB018 (slot 2) → found!
        ▼ 單一記憶體存取
```

**負載因子:** 取決於填滿程度
- Example 1: 1 插入 / 9 slot = 11% 使用率
- Example 2: 2 插入 / 7 slot = 29% 使用率
- Example 3: 1 插入 / 5 slot = 20% 使用率

### 鏈結法 (Chaining) — 假設實現

**假想實現:**
```c
typedef struct ChainNode {
    int key;
    int index;
    struct ChainNode* next;  // ← 指標追蹤,記憶體分散
} ChainNode;

typedef struct {
    ChainNode* buckets[size];
} ChainingHashTable;
```

**記憶體配置:**
```
0xB000 ┌─────────┐
       │ bucket0 │ ────▶ NULL
0xB008 │ bucket1 │ ────▶ 0xE000 ┌──────────┐
       │ bucket2 │       0xE000 │ node{k,i}│ ────▶ 0xE010
       │  ...    │              └──────────┘
0xB040 └─────────┘              0xE010 ┌──────────┐
                                       │ node{k,i}│ ────▶ NULL
                                       └──────────┘
       指標追蹤,記憶體分散到各處
```

| 特性 | 優點 | 缺點 |
|------|------|------|
| **記憶體配置** | 動態分配,無須預留 | 指標分散,快取不友善 |
| **快取效能** | 存取局部化 (同桶內) | 跨桶存取需指標追蹤 |
| **空間效率** | 可用低負載因子 | 指標開銷 (每節點 8 bytes) |
| **搜尋時間** | 平均 O(1), 最差 O(n) | 長鏈效能差 |
| **易於調整** | 簡單添加節點 | 需動態記憶體管理 |

### 效能對比表格

| 指標 | 開放定址法 | 鏈結法 |
|------|----------|--------|
| **時間複雜度 (平均)** | O(1) | O(1) |
| **時間複雜度 (最差)** | O(n) | O(n) |
| **快取局部性** | ★★★★★ (優秀) | ★★☆☆☆ (差) |
| **空間開銷** | 固定 (無指標) | 可變 (每項多 8 bytes) |
| **負載因子推薦** | < 0.7 | < 0.75 |
| **聚集風險** | 中等 | 無 |
| **實現複雜度** | 低 | 中 |

**Example 1 的記憶體對比:**

開放定址法:
```
雜湊表: 9 slot × 12 bytes/entry = 108 bytes
總計: 108 bytes (連續)
```

鏈結法 (假設平均 1 項/桶):
```
桶陣列: 9 bucket × 8 bytes/ptr = 72 bytes
節點: 1 個 × (4+4+8) bytes = 16 bytes
總計: 88 bytes (分散記憶體)
     但多處理指標追蹤,快取失效率高
```

## 7. 堆疊記憶體對比

### 目前實現 — 純迭代式

**程式碼骨架:**
```c
int* twoSum(int* nums, int numsSize, int* returnSize) {
    int tableSize = numsSize * 2 + 1;
    HashEntry* hashTable = (HashEntry*)malloc(sizeof(HashEntry) * tableSize);

    int* result = (int*)malloc(sizeof(int) * 2);

    for (int i = 0; i < numsSize; i++) {  // ← 單一迴圈,棧未深化
        int complement = target - nums[i];
        // 查詢、插入邏輯...
    }

    return result;
}
```

**堆疊使用:**
```
堆疊圖示 (函式堆疊框):
```

### 假設遞迴實現

**遞迴骨架 (反面教材):**
```c
bool findPair(int* nums, int numsSize, int target, int index,
              HashEntry* hashTable, int tableSize) {
    if (index == numsSize) {
        return false;
    }

    int complement = target - nums[index];
    // 查詢與插入...

    return findPair(nums, numsSize, target, index + 1,
                   hashTable, tableSize);  // ← 遞迴呼叫!
}
```

**堆疊使用 (遞迴版):**
```
堆疊圖示 (多層遞迴):
┌─────────────────────────────┐
│ findPair(..., index=4)      │ 第 5 層
│  返回地址, 參數, 區域變數    │
├─────────────────────────────┤
│ findPair(..., index=3)      │ 第 4 層
│  返回地址, 參數, 區域變數    │
├─────────────────────────────┤
│ findPair(..., index=2)      │ 第 3 層
│  返回地址, 參數, 區域變數    │
├─────────────────────────────┤
│ findPair(..., index=1)      │ 第 2 層
│  返回地址, 參數, 區域變數    │
├─────────────────────────────┤
│ findPair(..., index=0)      │ 第 1 層 (初始)
│  返回地址, 參數, 區域變數    │
├─────────────────────────────┤
│ main()                      │ 基礎
└─────────────────────────────┘

每層堆疊框: ~40-50 bytes (參數、返回地址、局部變數)
層數: numsSize (最差情況下所有元素)
總堆疊使用: ~40-50 × numsSize = O(numsSize) !!!

最差情況: numsSize=1000 時
堆疊深度: 1000 層
所需堆疊空間: ~40-50 KB
堆疊大小典型限制: 1-8 MB (視系統而定)
風險: 堆疊溢位 (Stack Overflow) ⚠️
```

**遞迴 vs 迴圈對比:**

| 特性 | 迴圈實現 | 遞迴實現 |
|------|--------|--------|
| **堆疊深度** | O(1) — 常數 | O(n) — 線性 |
| **堆疊風險** | 安全 | 可能溢位 |
| **函式呼叫開銷** | 無 | n 次呼叫 × 開銷 |
| **記憶體效率** | 優秀 | 差 |
| **可讀性** | 自然 | 可能更優雅 |
| **適用場景** | Two Sum (不適遞迴) | 樹遍歷、分治 |


## 結論

### 記憶體使用總結

**Example 1 完整記憶體組成:**

```
棧記憶體 (Stack):
├─ 返回地址:        8 bytes
├─ 舊 RBP:          8 bytes
├─ 局部變數:        ~40 bytes
└─ 小計:            ~56 bytes (O(1), 常數)

堆記憶體 (Heap):
├─ hashTable:       9 × 12 = 108 bytes
├─ result:          2 × 4 = 8 bytes
└─ 小計:            116 bytes (O(numsSize))

───────────────────────────────
總計:               ~172 bytes
```

**時間複雜度:**
- 平均情況: O(n) — 掃過陣列一次
- 最差情況: O(n²) — 線性探測聚集(罕見)

**空間複雜度:**
- 雜湊表: O(n)
- 棧: O(1)
- **總計: O(n)**

**假設地址注意:**
本文所有記憶體地址 (0xA000, 0xB000 等) 為假設值,僅用於演示。實際執行時地址由作業系統動態分配,不應依賴於特定地址值。
