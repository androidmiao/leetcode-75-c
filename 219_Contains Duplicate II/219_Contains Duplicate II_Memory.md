# LeetCode 219 Contains Duplicate II — 記憶體分析

## 1. Struct 記憶體佈局

### HashMap 結構定義
```c
typedef struct {
    int* keys;      // 指向動態分配的陣列，儲存視窗內的元素值
    int* values;    // 指向動態分配的陣列，儲存對應的索引
    int size;       // 目前有效元素數量
    int capacity;   // 最大容量（k）
} HashMap;
```

### 欄位大小與偏移量
```
┌────────────────────────────────────┐
│ HashMap 結構內部佈局               │
├────────────────────────────────────┤
│ 位移      │ 欄位名稱  │ 類型     │
├───────────┼───────────┼──────────┤
│ 0x00      │ keys      │ int*     │  8 bytes (指標)
│ 0x08      │ values    │ int*     │  8 bytes (指標)
│ 0x10      │ size      │ int      │  4 bytes
│ 0x14      │ capacity  │ int      │  4 bytes (對齊填充)
├────────────────────────────────────┤
│ 結構總大小：24 bytes               │
│ 對齐：8 bytes (指標大小)           │
└────────────────────────────────────┘
```

### 假設記憶體位置圖示
```
假設 HashMap 實例位置：0xA000

0xA000  ┌─────────────────────┐
        │  keys (指標)        │ ───────▶ (指向 0xB000 的 int 陣列)
        ├─────────────────────┤
0xA008  │  values (指標)      │ ───────▶ (指向 0xC000 的 int 陣列)
        ├─────────────────────┤
0xA010  │  size (int)         │ = 0 (初始化時)
        ├─────────────────────┤
0xA014  │  capacity (int)     │ = k (由調用者傳入)
        │ (4-byte 填充)       │
        └─────────────────────┘
```

## 2. 完整資料結構記憶體配置

### 整體記憶體佈局圖
```
棧 (Stack)                        堆 (Heap)
┌─────────────────────┐         ┌──────────────────────────────────┐
│ HashMap @ 0xA000    │         │ keys[] 陣列 @ 0xB000             │
├─────────────────────┤         ├──────────────────────────────────┤
│ keys     ──────────────────▶  │ [0]: int (4 bytes)               │
│ @ 0xA000 │ 0xB000  │         │ [1]: int (4 bytes)               │
│          ├─────────┤         │ [2]: int (4 bytes)               │
│ values   ──────────────────▶  │ ...                              │
│ @ 0xA008 │ 0xC000  │         │ [k-1]: int (4 bytes)             │
│          ├─────────┤         └──────────────────────────────────┘
│ size     │   0     │
│ @ 0xA010 │ (初)    │         ┌──────────────────────────────────┐
│          ├─────────┤         │ values[] 陣列 @ 0xC000           │
│ capacity │   k     │         ├──────────────────────────────────┤
│ @ 0xA014 │         │         │ [0]: int (4 bytes)               │
│          └─────────┘         │ [1]: int (4 bytes)               │
└─────────────────────┘         │ [2]: int (4 bytes)               │
                                │ ...                              │
                                │ [k-1]: int (4 bytes)             │
                                └──────────────────────────────────┘

記憶體分配：
• HashMap 結構體：24 bytes (棧)
• keys[] 陣列：k × 4 bytes (堆)
• values[] 陣列：k × 4 bytes (堆)
• 總堆空間：8k bytes
```

## 3. 陣列索引版對應

### 平行陣列映射關係

此實現使用**平行陣列（Parallel Arrays）**技術，而非傳統的鏈式雜湊表：

```
邏輯視圖 (視窗內容)              物理視圖 (陣列儲存)
─────────────────────────────────────────────────

元素 #1: 值=1, 索引=0          keys[0] = 1
元素 #2: 值=2, 索引=1          values[0] = 0
元素 #3: 值=3, 索引=2          ────────────

                               keys[1] = 2
                               values[1] = 1
                               ────────────

                               keys[2] = 3
                               values[2] = 2
                               ────────────

對應關係：
┌────────────────────────────────────────────────┐
│ size = 3, capacity = 4 (k=4)                   │
├─────────┬──────────┬──────────┬──────────┐
│ Index   │ [0]      │ [1]      │ [2]      │
├─────────┼──────────┼──────────┼──────────┤
│ keys[]  │ 1        │ 2        │ 3        │
│ values[]│ 0        │ 1        │ 2        │
├─────────┴──────────┴──────────┴──────────┤
│ 第一個有效元素始終在 keys[0]、values[0]  │
│ 最後一個有效元素在 keys[size-1]、       │
│ values[size-1]                           │
└────────────────────────────────────────────┘
```

### 搜尋流程
```
給定要搜尋的值 num：
1. 線性掃描 keys[] 陣列，i = 0 to size-1
2. 若 keys[i] == num，得到結果
   - 現在索引：i_current
   - 歷史索引：values[i]
   - 索引差值：i_current - values[i]
   - 決定：若差值 <= k，返回 true；否則繼續

優點：
  • 快取親和性：連續記憶體訪問
  • 簡單實現：無複雜的雜湊碰撞處理

缺點：
  • O(k) 搜尋複雜度（不是 O(1)）
  • 但由於 k 通常小，仍可接受
```

| 5 | 移出 & 插入 | 4 | 2 | [1, 2, ?] | [3, 4, ?] | 2 | 無匹配 |
| 6 | 移出 & 插入 | 5 | 3 | [2, 3, ?] | [4, 5, ?] | 2 | **false** |

**記憶體檢查點：** 步驟 3-6 中，視窗寬度始終 ≤ k=2，所有重複元素的索引差 > k

## 7. 退化情況分析

### 最壞情況：所有元素不同 & k 較大

```
假設：nums = [1, 2, 3, 4, 5, ..., N]，k = 1000，所有元素不同

初始化：
  • HashMap 結構：24 bytes
  • keys[] 陣列：1000 × 4 = 4000 bytes
  • values[] 陣列：1000 × 4 = 4000 bytes
  ─────────────────────────────
  總計：8024 bytes（堆）

執行流程：
┌─────────────────────────────────────────────────────┐
│ 步驟 i   │ num = nums[i] │ 操作      │ 視窗大小    │
├─────────────────────────────────────────────────────┤
│ 0-999    │ 各不相同      │ 順序插入  │ 0 → 1000    │
│ 1000+    │ 各不相同      │ 移出&插入 │ 維持 1000   │
├─────────────────────────────────────────────────────┤
│ 每次搜尋  │ -             │ 線性掃描  │ O(k)=O(1000)│
│          │               │ keys[]    │ 成本高      │
└─────────────────────────────────────────────────────┘

記憶體峰值：
  • HashMap：24 bytes
  • keys[]：4 KB
  • values[]：4 KB
  ─────────────────────────────
  總峰值：約 8.024 KB

時間複雜度退化：
  • n 次迭代 × 每次 O(k) 搜尋 = O(n × k)
  • 若 k = 1000，大陣列會更快耗盡 CPU 快取資源
  • 較差的快取局部性（特別是在邊界搜尋時）

比較：若用雜湊表
  • 相同的記憶體用於雜湊桶
  • 搜尋時間 O(1) 平均，不受 k 影響
  • 但節點分配產生額外開銷（指標、元資料）

最佳化機會：
  1. 若 k 超大，考慮使用標準雜湊表或樹結構
  2. 使用二分搜尋而非線性掃描（但前提是 keys[] 有序）
     — 當前實現無序，故不可行
  3. 若 k >> N，使用 HashSet 取代視窗
```

## 結論與最佳實踐

### 記憶體管理要點

1. **平行陣列方案的優勢**
   - 簡潔的雙指標對應
   - 快取親和性強（連續記憶體）
   - 適合小 k 值（< 1000）

2. **指標追蹤關鍵**
   - keys[] 和 values[] 始終同步
   - 一個操作更新兩個陣列
   - 索引差計算依賴 values[]

3. **記憶體訪問順序**
   ```
   掃描 keys[0], keys[1], ..., keys[size-1]
   同時訪問 values[0], values[1], ..., values[size-1]
   → 兩個陣列的快取命中率高
   ```

4. **邊界條件**
   - size < capacity：直接插入（O(1)）
   - size == capacity：移出舊元素並插入新元素
   - 搜尋過程始終掃描 [0, size)

5. **長期維護**
   - 無動態調整（容量固定 k）
   - 無碎片化風險
   - 預測性能開銷：O(n × k) 搜尋 + O(n) 插入
