# LeetCode 1926 「迷宮最近出口」BFS 記憶體佈局分析

## 1. Struct 記憶體佈局

此問題不使用結構體，但需追蹤的關鍵變數有：

### 主要變數定義
```c
int queueRows[10000];     // 佇列行座標陣列
int queueCols[10000];     // 佇列列座標陣列
int directions[4][2];     // 四個方向移動表
int front;                // 佇列讀取指標
int rear;                 // 佇列寫入指標
int steps;                // 距離計數
int rows, cols;           // 迷宮尺寸
int startRow, startCol;   // 起點座標
```

### 各變數大小與棧配置

```
假設地址基數：0xA000（棧空間的示意位址）

變數名稱          型別        大小      記憶體位址      備註
───────────────────────────────────────────────────────────
front             int         4 bytes   0xA000        讀取指標
rear              int         4 bytes   0xA004        寫入指標
steps             int         4 bytes   0xA008        步數計數
rows              int         4 bytes   0xA00C        迷宮列數
cols              int         4 bytes   0xA010        迷宮行數
startRow          int         4 bytes   0xA014        起點列
startCol          int         4 bytes   0xA018        起點行
───────────────────────────────────────────────────────────

陣列配置（棧空間）

queueRows[10000]  int[10000]  40000 B   0xA100-       平行陣列 #1
queueCols[10000]  int[10000]  40000 B   0xA100-       平行陣列 #2
                                        (分開存放)

directions[4][2]  int[4][2]   32 bytes  0xC100-       4 方向表
                                        0xC11F
```

## 2. 完整資料結構記憶體配置

### 平行陣列佇列的記憶體佈局

```
queueRows 與 queueCols 是兩個獨立的記憶體區塊，
但邏輯上配對使用。

假設 queueRows 基地址 = 0xB000 (40000 bytes)
    queueCols 基地址 = 0xC2A0 (40000 bytes)

時間点 1：初始狀態，入口 [1,2] 加入
─────────────────────────────────────────────

queueRows 記憶體視圖：
┌──────────┬──────────┬──────────┬──────────┬───────────┐
│ Index 0  │ Index 1  │ Index 2  │ Index 3  │    ...    │
├──────────┼──────────┼──────────┼──────────┼───────────┤
│  [1]     │  [?]     │  [?]     │  [?]     │  [?]      │
└──────────┴──────────┴──────────┴──────────┴───────────┘
  0xB000     0xB004     0xB008     0xB00C

queueCols 記憶體視圖：
┌──────────┬──────────┬──────────┬──────────┬───────────┐
│ Index 0  │ Index 1  │ Index 2  │ Index 3  │    ...    │
├──────────┼──────────┼──────────┼──────────┼───────────┤
│  [2]     │  [?]     │  [?]     │  [?]     │  [?]      │
└──────────┴──────────┴──────────┴──────────┴───────────┘
  0xC2A0     0xC2A4     0xC2A8     0xC2AC

指標狀態：
  front = 0  ──▶ queueRows[0]=1, queueCols[0]=2  ← 讀位置
  rear  = 1  ──▶ 下一個寫入位置
```

### 平行陣列佇列詳細圖示

```
邏輯上的佇列結構：

    第 0 筆元素        第 1 筆元素
   ┌──────────┐      ┌──────────┐
   │ row=1    │      │ row=?    │
   │ col=2    │      │ col=?    │
   └──────────┘      └──────────┘
      ↑                   ↑
   queueRows[0]       queueRows[1]
   queueCols[0]       queueCols[1]


時間点 2：處理第 1 層 BFS 後，[0,2] 和 [1,1] 加入
─────────────────────────────────────────────────────

queueRows：
┌──────────┬──────────┬──────────┬──────────┐
│ Index 0  │ Index 1  │ Index 2  │ Index 3  │
├──────────┼──────────┼──────────┼──────────┤
│  [1]     │  [0]     │  [1]     │  [?]     │ ← 舊資料
└──────────┴──────────┴──────────┴──────────┘
  0xB000     0xB004     0xB008     0xB00C
  (已出隊)    ↑          ↑
           front      rear

指標位置：
  front = 1  ──▶ 下一個要出隊的位置
  rear  = 3  ──▶ 下一個可寫入的位置

邏輯上有效的佇列元素：
  index 1: (0, ?)
  index 2: (1, ?)
```

### 迷宮 char** 的記憶體佈局

```
char** maze 是一個二級指標，指向行指標陣列。

假設：
  maze 的基地址 @ 0xD000  (指標本身)
  maze[0] 指向   0xE000   (第 0 列的字符陣列)
  maze[1] 指向   0xE100   (第 1 列的字符陣列)
  maze[2] 指向   0xE200   (第 2 列的字符陣列)

舊值 → 新值轉變：

迷宮初始狀態（Example 1）：
  maze[0] = "++-+"
  maze[1] = "...+"
  maze[2] = "+++-"

記憶體圖：

┌─ 0xD000 （maze 指標陣列基地址）
│ ┌─────────┬─────────┬─────────┐
│ │ 0xE000  │ 0xE100  │ 0xE200  │ ← 三個列指標
│ └─────────┴─────────┴─────────┘
│    ↓        ↓        ↓
│  列 0      列 1      列 2
│
├─ 0xE000 （第 0 列字符陣列）
│ ┌──┬──┬──┬──┐
│ │'+' │'+' │'.' │'+' │
│ └──┴──┴──┴──┘
│  [0] [1] [2] [3]
│
├─ 0xE100 （第 1 列字符陣列）
│ ┌──┬──┬──┬──┐
│ │'.' │'.' │'E'(→'+') │'+' │
│ └──┴──┴──┴──┘
│  [0] [1] [2] [3]
│           ↑ 標記為已訪問
│
└─ 0xE200 （第 2 列字符陣列）
  ┌──┬──┬──┬──┐
  │'+' │'+' │'+' │'.' │
  └──┴──┴──┴──┘
   [0] [1] [2] [3]

訪問模式：
  maze[1][2] = '.' (入口)
              ↓
              '+' (標記為已訪問)
```

## 3. 陣列索引版對應

### 佇列索引到座標的映射

```
關鍵關係式：
  第 i 個佇列元素 = (queueRows[i], queueCols[i])
                  ↓
                 迷宮座標
                  ↓
              maze[row][col]

索引對應表（假設佇列已進行到 front=1, rear=3）：

佇列索引  queueRows  queueCols   座標        迷宮內容    狀態
───────────────────────────────────────────────────────
  0         1         2        (1,2)      maze[1][2]   已出隊
  1         0         2        (0,2)      maze[0][2]   ← front
  2         1         1        (1,1)      maze[1][1]   ← rear-1
  3         [?]       [?]       -           -          未用
```

### 記憶體位址到迷宮座標的追蹤

```
例子：對 front=1 進行讀取

步驟 1：讀行座標
  int row = queueRows[front];
           = queueRows[1]
           ← 從 0xB000 + 1*4 = 0xB004 讀取
           = 0

步驟 2：讀列座標
  int col = queueCols[front];
           = queueCols[1]
           ← 從 0xC2A0 + 1*4 = 0xC2A4 讀取
           = 2

步驟 3：訪問迷宮
  maze[row][col] = maze[0][2]
                 ← 根據 row、col 計算出位址
                 = *(maze[0] + 2)
                 = *(0xE000 + 2)
                 = 0xE002
                 讀出字符 '.'

步驟 4：遞增前指標
  front++;
          ← front 從 0xA000 的值改為 2
```

## 4. 搜尋／操作過程追蹤

### Example 1 完整追蹤

迷宮配置：
```
maze = [
  ["+", "+", ".", "+"],
  [".", ".", ".", "+"],
  ["+", "+", "+", "."]
]
entrance = [1, 2]

座標圖：
     0   1   2   3
  ┌─────────────────┐
0 │ + │ + │ . │ + │
1 │ . │ . │ E │ + │
2 │ + │ + │ + │ . │
  └─────────────────┘
```

#### 初始化階段

```
程式執行：
  queueRows[0] = 1;
  queueCols[0] = 2;
  rear++;
  maze[1][2] = '+';

記憶體狀態變更：

舊值 ────▶ 新值
──────────────────────────────────

front: 0 ──▶ 0 (不變)
rear:  0 ──▶ 1

queueRows[0]: [未初始化] ──▶ [1]  @ 0xB000
queueCols[0]: [未初始化] ──▶ [2]  @ 0xC2A0

maze[1][2]: '.' ──▶ '+'  @ 0xE102

指標移動摘要：
┌────────────────────┐
│ rear: 0 ──────▶ 1  │
│ front: 0 (不變)    │
└────────────────────┘

邏輯佇列狀態：
[ (1,2) ]
  ↑
front/rear-1
```

#### 進入 while 迴圈第 1 次

```
計算層大小：
  levelSize = rear - front = 1 - 0 = 1
  steps++ : steps = 1

訪問位址：
  row = queueRows[0]  ← 位址 0xB000 讀取值 1
  col = queueCols[0]  ← 位址 0xC2A0 讀取值 2

記憶體操作：
┌─────────────────────────────────────────┐
│ step  front  rear  訪問位址    讀值  操作 │
├─────────────────────────────────────────┤
│ 1     0      1    0xB000      1    LD  │
│       0      1    0xC2A0      2    LD  │
│       1      1    (後)              INC│
└─────────────────────────────────────────┘

指標移動摘要：
┌────────────────────────────┐
│ front: 0 ──────▶ 1         │
│ rear: 1 (不變)             │
└────────────────────────────┘
```

#### 檢查四個方向

```
從 (1,2) 檢查鄰居：

方向 0 (下)：newRow=2, newCol=2
  位址計算：maze[2][2] = *(0xE200 + 2) = 0xE202
  讀值：'+'  (牆壁，跳過)

方向 1 (上)：newRow=0, newCol=2
  位址計算：maze[0][2] = *(0xE000 + 2) = 0xE002
  讀值：'.'  (空地，檢查邊界)
  邊界檢查：row == 0 ✓ (在第 0 列，是邊界)
  ➜ 找到出口！直接 return 1
```

迷宮配置：

#### 初始化與第 1 層

```
初始：
  queueRows[0] = 1
  queueCols[0] = 0
  maze[1][0] = '+'

記憶體變更 (舊 ──▶ 新)：
┌─────────────────────────────────┐
│ front: 0 ──▶ 0 (不變)           │
│ rear:  0 ──▶ 1                 │
│ steps: 0 ──▶ 0 (初始)          │
│                                │
│ queueRows[0]: ? ──▶ 1 @ 0xB000 │
│ queueCols[0]: ? ──▶ 0 @ 0xC2A0 │
│ maze[1][0]: '.' ──▶ '+'        │
└─────────────────────────────────┘

while 第 1 次：
  levelSize = 1 - 0 = 1
  steps = 1

  取出 (1,0)：
    row = queueRows[0] = 1 @ 0xB000
    col = queueCols[0] = 0 @ 0xC2A0
    front++: front = 1

  檢查四個方向：
    下 (2,0)：maze[2][0] = '+' (牆，跳過)
    上 (0,0)：maze[0][0] = '+' (牆，跳過)
    右 (1,1)：maze[1][1] = '.' (空地，非邊界)
             ➜ 標記並加入
             maze[1][1] = '+'
             queueRows[1] = 1 @ 0xB004
             queueCols[1] = 1 @ 0xC2A4
             rear++: rear = 2
    左 (1,-1)：越界，跳過

  第 1 層結束狀態：
  ┌─────────────────────┐

  │ rear: 2             │

  └─────────────────────┘
```

#### 第 2 層

```
while 第 2 次：
  levelSize = 2 - 1 = 1
  steps = 2

  取出 (1,1)：
    row = queueRows[1] = 1 @ 0xB004
    col = queueCols[1] = 1 @ 0xC2A4
    front++: front = 2

  檢查四個方向：
    下 (2,1)：maze[2][1] = '+' (牆，跳過)
    上 (0,1)：maze[0][1] = '+' (牆，跳過)
    右 (1,2)：maze[1][2] = '.' (空地，檢查邊界)
             newCol == 2, cols-1 == 2
             ✓ 在邊界！且是空地！
             ➜ return steps = 2
    左 (1,0)：已拜訪 (已改為 '+')，跳過

  最終結果：return 2
```

## 6. 記憶體存取模式比較

### 平行陣列 vs 結構化佇列

#### 方案 A：平行陣列（本解答採用）

```c
int queueRows[10000];
int queueCols[10000];
int front, rear;
```

**記憶體配置：**
```
queueRows: 連續 40000 bytes
queueCols: 連續 40000 bytes
共 80000 bytes

記憶體視圖：
┌─────────────────────┐
│   queueRows[0..9999]│ (連續)
│   @0xB000-0x9C3F   │
├─────────────────────┤
│   queueCols[0..9999]│ (連續)
│   @0xC2A0-0xD263   │
└─────────────────────┘

快取友好度：★★★☆☆
  - 每次讀取 queueRows[i]，會預讀多個索引
  - 但訪問 queueCols[i] 時，需跳過 40KB，快取未命中
  - 雙陣列訪問有空間局部性差
```

**優點：**
- 記憶體連續，掃描時快取預取率高
- 對簡單 for 迴圈的向量化優化友好
- 節省對齊填充（無結構體開銷）

**缺點：**
- 快取行汙染（訪問 rows 和 cols 需跳躍）
- 語義不夠清晰（座標被分散）
- 搜索時需同時索引兩個陣列

---

#### 方案 B：結構化佇列

```c
struct Node {
  int row;    // 4 bytes
  int col;    // 4 bytes
};  // 8 bytes 對齐

struct Node queue[10000];
int front, rear;
```

**記憶體配置：**
```
queue[0..9999]: 80000 bytes (連續)
  queue[0]: row(0x00-0x03), col(0x04-0x07)
  queue[1]: row(0x08-0x0B), col(0x0C-0x0F)
  ...

記憶體視圖：
┌───────────────────────────────┐
│ queue[0]: [row][col]          │ 8 bytes
│ queue[1]: [row][col]          │ 8 bytes
│ queue[2]: [row][col]          │ 8 bytes
│ ...                           │
└───────────────────────────────┘

快取友好度：★★★★★
  - row 和 col 相鄰，同一快取行
  - 讀取 queue[i] 時自動拿到 row 和 col
  - 空間局部性最優
```

**優點：**
- 快取效率最高（相鄰訪問）
- 語義清晰，易讀易維護
- CPU 可能做向量化優化

**缺點：**
- 需要結構體定義開銷
- 填充對齐可能浪費空間（此例無）
- 代碼稍多

---

#### 性能對比

```
操作            平行陣列                  結構化佇列
────────────────────────────────────────────────
讀取 queue[i]   queueRows[i]  +           struct {r,c}
                queueCols[i]  (兩次存取)   (一次存取)

快取未命中率    ~15-20%（跳躍存取）     ~3-5%（相鄰存取）

寫入 queue[i]   rear 時需兩次寫          一次寫
                (queueRows, queueCols)

代碼複雜度      低（簡單陣列）          中等（結構體）

BFS 100x100網格  ~250ms (平行)           ~180ms (結構化)
運行時間        (估計，硬體相關)
```

### 訪問模式vs內存配置

#### 平行陣列的記憶體存取序列

```
時刻   操作           位址          快取行    快取命中
──────────────────────────────────────────────────
t0  queueRows[0] <- 0xB000        Line A    Miss
    (預取 0xB000-0xB03F)
t1  queueCols[0] <- 0xC2A0        Line B    Miss
    (預取 0xC2A0-0xC2DF)
t2  queueRows[1] <- 0xB004        Line A    Hit
t3  queueCols[1] <- 0xC2A4        Line B    Hit
...

快取狀況：
  L1 Cache (32KB): 存不下兩個陣列頭部
  L2 Cache (256KB): 放下 ~6.4 個 queueRows 區塊
  跳躍訪問導致頻繁置換
```

#### 結構化佇列的記憶體存取序列

```
時刻   操作              位址              快取行    快取命中
────────────────────────────────────────────────────────
t0  queue[0].row <- 0xD000            Line C    Miss
    queue[0].col <- 0xD004            Line C    Hit
    (預取 0xD000-0xD03F)
t1  queue[1].row <- 0xD008            Line C    Hit
    queue[1].col <- 0xD00C            Line C    Hit
t2  queue[2].row <- 0xD010            Line C    Hit
...

快取狀況：
  L1 Cache (32KB): 可以完全存下 4000 個元素
  L2 Cache (256KB): 可存下全部佇列
  順序訪問，快取命中率接近 99%
```

### 原地迷宮標記 vs 獨立訪問陣列

#### 方案 A：原地標記（本解答採用）

```c
maze[startRow][startCol] = '+';  // 入口標記為牆
maze[newRow][newCol] = '+';      // 訪問過的改為牆
```

**優點：**
- 無額外記憶體：O(1) 額外空間
- 隱式訪問狀態，無需維護單獨陣列
- 遍歷快：只需讀寫迷宮本身

**缺點：**
- 破壞輸入（函數有副作用）
- 無法復原迷宮（題目無所謂）
- 多線程不安全

**記憶體使用：**
```
maze: m × n （題目輸入）
額外: 0 bytes
┌─────────────────────┐
│ maze[0][0] = '+'    │ (原位改)
│ maze[0][1] = '.'    │
│ ...                 │
└─────────────────────┘
總計：O(m*n)
```

---

#### 方案 B：獨立訪問陣列

```c
bool visited[m][n];  // 或 char visited[m][n]
visited[row][col] = true;
```

**優點：**
- 不修改輸入
- 可多次跑 BFS
- 更規範的演算法設計

**缺點：**
- 額外 m × n 字節
- 需初始化 visited 陣列
- 兩倍的記憶體存取（讀迷宮 + 讀 visited）

**記憶體使用：**
```
maze: m × n （題目輸入）
visited: m × n （新陣列）
額外：m * n bytes

示例（100x100 迷宮）：
  原方案：100*100 chars = 10KB
  新方案：10KB (maze) + 10KB (visited) = 20KB
  多 100%
```

**性能對比**：

```
操作              原地標記          獨立訪問陣列
────────────────────────────────────────────
檢查是否訪問      maze[r][c] != '.'   visited[r][c]
                (1次讀)            (1次讀)

標記已訪問        maze[r][c] = '+'    visited[r][c] = 1
                (1次寫)            (1次寫)

額外記憶體        0                m * n

快取效率          更高               稍低
                (集中在 maze)      (兩個陣列)

時間複雜度        O(m*n)            O(m*n)
空間複雜度        O(min(m*n, Q))    O(m*n)
```

## 7. 堆疊記憶體對比

### 迭代 BFS（本解答） vs 遞迴 DFS

#### 方案 A：迭代 BFS（本解答）

```c
int queueRows[10000], queueCols[10000];
int front = 0, rear = 0;

// 使用堆上的靜態陣列（全局或棧上的大陣列）
```

**記憶體使用：**
```
棧空間：
  frame (nearestExit):
    - local 變數 ~60 bytes
    - queueRows[10000]: 40000 bytes
    - queueCols[10000]: 40000 bytes
    ─────────────────────────
    總計：~80060 bytes

堆空間：0 (無動態配置)

最大深度：1 frame (非遞迴)
```

**棧配置圖：**
```
棧頂 (棧指標 esp/rsp)
    ┌──────────────────┐
    │ 局部變數         │ ~60 bytes
    │ (front,rear,step)│
    ├──────────────────┤
    │ queueRows[10000] │ 40000 bytes
    │ (連續記憶體)     │
    ├──────────────────┤
    │ queueCols[10000] │ 40000 bytes
    │ (連續記憶體)     │
    ├──────────────────┤
    │ 返回位址         │ 8 bytes (64-bit)
    │ (main -> frame)  │
棧底 └──────────────────┘
```

**優點：**
- 堆棧深度固定，最多 1 層
- 記憶體配置靜態，無動態分配開銷
- 空間效率高

**缺點：**
- 棧陣列很大（80KB），可能棧溢位（棧通常 1-8MB）
- 代碼不如遞迴直觀

---

#### 方案 B：遞迴 DFS

```c
void dfs(char** maze, int row, int col, int steps, ...) {
  // 基礎情況檢查
  if (isExit(row, col)) {
    return steps;
  }

  // 遞迴呼叫四個方向
  for (int d = 0; d < 4; d++) {
    int nr = row + directions[d][0];
    int nc = col + directions[d][1];
    if (isValid(nr, nc)) {
      dfs(maze, nr, nc, steps+1, ...);
    }
  }
}
```

**記憶體使用：**
```
遞迴深度 k 時的堆棧：

main
├─ nearestExit
├─ dfs(frame 1)
├─ dfs(frame 2)
├─ ...
└─ dfs(frame k) ← 最大深度

每個 dfs frame：
  - 參數（指標）：~40 bytes
  - 局部變數：~20 bytes
  - 返回位址：8 bytes
  ──────────────
  每層：~68 bytes

最大深度：迷宮對角線 = sqrt(m² + n²)
  例：100x100 迷宮 → 深度 ~141 層

最大棧用量：141 * 68 ≈ 9.6 KB
  (遠小於 BFS 的 80KB)
```

**棧配置圖（深度 k）：**
```
棧頂
    ┌─────────────────┐
    │ dfs(frame k)    │ 68 bytes
    │ local var       │
    │ return addr     │
    ├─────────────────┤
    │ dfs(frame k-1)  │ 68 bytes
    ├─────────────────┤
    │ ...             │
    ├─────────────────┤
    │ dfs(frame 1)    │ 68 bytes
    ├─────────────────┤
    │ nearestExit     │ ~100 bytes
棧底 └─────────────────┘

總深度：141 層 × 68 bytes ≈ 9.6 KB
```

### 效能對比：BFS vs DFS

```
特性              BFS 迭代              DFS 遞迴
────────────────────────────────────────────
棧使用量        ~80 KB (靜態陣列)      ~10 KB (幀棧)
堆使用量        0                     0
最大棧深度      1 層                  m√n 層

時間複雜度      O(m*n)                O(m*n)
空間複雜度      O(m*n)                O(m√n)

棧溢位風險      高                    低
  (大陣列)      (1/8MB 棧)           (深度 141)

代碼複雜度      中等                  簡單
                (手動 queue)          (系統 call stack)

最短路徑保證    ✓ (層級保證)         ✗ (先探深度)
  (此題要求)

快取效率        ★★★☆☆              ★★★★☆
                (跳躍存取)            (遞迴呼叫順序)
```

### 100×100 迷宮的具體數字

```
迷宮大小：100 × 100 = 10000 格

最壞情況（全可走路）：
  BFS 佇列最大 = 10000 個節點
              = 10000 * 8 bytes = 80KB

  DFS 遞迴深度 = sqrt(100² + 100²) ≈ 141
              = 141 * 68 bytes ≈ 9.6KB

BFS 棧配置：
  queueRows[10000]:  40KB
  queueCols[10000]:  40KB
  局部變數：         ~0.1KB
  ────────────────
  合計：             ~80.1KB

DFS 棧配置：
  frame 層數：       141
  每層 68 bytes：    9.5KB
  ────────────────
  合計：             ~9.6KB

結論：
  BFS 棧需求 ÷ DFS 棧需求 ≈ 80KB ÷ 10KB = 8 倍
  但 BFS 更適合此題（求最短路徑）
  DFS 需追蹤全局最小值，較複雜
```

---

### 為何選擇 BFS

```
❶ 最短路徑保證
   BFS 一層一層展開，第一次遇到出口必是最短

❷ 時間複雜度相同
   O(m*n) 都要訪問全部格子

❸ 空間複雜度優勢（在題目情境下）
   - BFS: O(min(m*n, Q)) Q=佇列大小
   - DFS: O(m√n) 遞迴深度

   實際上 BFS 在大迷宮中佇列不會滿
   (出口通常較快找到)

❹ 代碼直觀性
   對此類單源最短路，BFS 標準解法
```

## 8. 總結與最佳實踐

### 記憶體優化檢查表

```
✓ 使用平行陣列而非結構化佇列
  原因：省去對齐填充，訪問模式簡單

✓ 原地標記迷宮而非獨立 visited 陣列
  原因：節省 O(m*n) 額外記憶體

✓ 靜態棧陣列（大小 10000）而非動態分配
  原因：避免 malloc/free 開銷

✓ 使用迭代 BFS 而非遞迴 DFS
  原因：最短路徑保證，記憶體管理簡單

✓ 指標運算而非二維陣列索引（高級優化）
  原因：CPU 快取友好，避免二度尋址
  (此例中未採用，保留可讀性)
```

### 指標變數的生命週期

```
變數名      初值  變化軌跡                      最終值
───────────────────────────────────────────────
front       0    0→1→2→...→rear           rear

rear        0    0→1→2→3→...→m*n          m*n
                 (依照發現節點數)          (最多)

steps       0    0→1→2→...→最短路徑        返回值

queueRows   -    逐次填入 row 座標         (部分填詢)
queueCols   -    逐次填入 col 座標         (部分填詢)

maze[][]    ./'+ 依訪問改為 '+'            全為 '+' 或 '.'
            (入) (出)                     (無出口時全改)
```
