# LeetCode 328：奇偶鏈表 - 記憶體分析

## 1. Struct 記憶體佈局

### ListNode 結構定義
```c
struct ListNode {
    int val;              // offset 0, size 4 bytes
    struct ListNode* next; // offset 8, size 8 bytes (64-bit)
};
```

### 記憶體佈局圖（64位系統）

```
┌─────────────┬──────────┬──────────────────┐
│   val (int) │ padding  │  next (pointer)  │
│   4 bytes   │ 4 bytes  │    8 bytes       │
├─────────────┼──────────┼──────────────────┤
│ offset 0    │ offset 4 │   offset 8       │
└─────────────┴──────────┴──────────────────┘
                    總計：16 bytes
```

### 單一節點記憶體示意

```
    地址 0xA000
    ┌────────────────────────────┐
    │   ListNode 節點            │
    ├────────────────────────────┤
    │ val: 5          [4 bytes]  │  offset 0 ~ 3
    │ padding         [4 bytes]  │  offset 4 ~ 7
    │ next: 0x0000... [8 bytes]  │  offset 8 ~ 15
    └────────────────────────────┘
         總計：16 bytes
```

## 3. 搜尋／操作過程追蹤

### Example 1 詳細追蹤：[1,2,3,4,5]

#### 初始狀態
```
    odd              even           starting_even
    0xA000 (val=1)   0xA010 (val=2) 0xA010 (const)
    │                │              │
    ▼                ▼              ▼
    [1] ──▶ [2] ──▶ [3] ──▶ [4] ──▶ [5] ──▶ NULL
```

#### 迭代 Step 1

**操作流程：**
```
1. odd->next = even->next = 0xA020(node3)
   └─ node1.next 從 0xA010 改為 0xA020

2. odd 前進至 even->next (node3)
   └─ odd = 0xA020

3. even->next = odd->next = 0xA030(node4)
   └─ node2.next 從 0xA020 改為 0xA030

4. even 前進至 odd->next (node4)
   └─ even = 0xA030
```

**記憶體狀態轉換：**

```
操作前：
[1]──▶[2]──▶[3]──▶[4]──▶[5]──▶NULL
 ▲     ▲
odd   even

操作後：
[1]──▶[3]──▶[4]──▶[5]──▶NULL
       ▲     ▲
      odd   even
[2]───┘
```

#### 迭代 Step 2

**操作流程：**
```
1. odd->next = even->next = NULL
   └─ node3.next 從 0xA040 改為 0x0(NULL)

2. odd 前進至 even->next (NULL)
   └─ odd = NULL

3. 檢查：odd->next == NULL → break
```

**記憶體狀態轉換：**

```
操作後：
[1]──▶[3]──▶[5]──▶NULL

[2]──▶[4]──▶NULL
```

#### 最後連接

```
odd->next = starting_even (0xA010)
└─ node5.next = 0xA010

最終鏈表結構：
[1] ──▶ [3] ──▶ [5] ──▶ [2] ──▶ [4] ──▶ NULL
0xA000  0xA020  0xA040  0xA010  0xA030
```

```
  even = 0xB030(val=5)

狀態：
[2] ──▶ [3] ──▶ [5] ──▶ [6] ──▶ [4] ──▶ [7] ──▶ NULL
        ▲              ▲
       odd           even

[1] ──▶ [5] ...
```

```
操作：
  odd->next = even->next = 0xB040(node5)  → node3.next = 0xB040

狀態：
[2] ──▶ [3] ──▶ [6] ──▶ [4] ──▶ [7] ──▶ NULL
                ▲           ▲

[1] ──▶ [5] ──▶ [4] ...
```

#### 迭代 Step 3

```
操作：
  odd->next = even->next = NULL → node6.next = NULL
  odd = NULL → break

狀態：
[2] ──▶ [3] ──▶ [6] ──▶ NULL

[1] ──▶ [5] ──▶ [4] ──▶ [7] ──▶ NULL
```

#### 最後連接

```
odd->next = starting_even (0xB010)
→ node6.next = 0xB010

最終結果：
[2] ──▶ [3] ──▶ [6] ──▶ [7] ──▶ [1] ──▶ [5] ──▶ [4] ──▶ NULL

```

**驗證：**
- 奇數索引位置 (0,2,4,6)：2 → 3 → 6 → 7 ✓

## 5. 記憶體存取模式比較

### 鏈表 (Linked List) vs 陣列 (Array) 方法

#### 資料結構佔用空間

| 項目 | 鏈表方法 | 陣列方法 |
|------|--------|--------|
| 單節點大小 | 16 bytes (4 + 4 padding + 8 pointer) | 4 bytes (int) |
| n=5 總空間 | 16 × 5 = 80 bytes | 4 × 5 = 20 bytes |
| 額外空間 | 8 bytes × n (next指標) | 0 bytes (原地操作) |
| 空間複雜度 | O(n) (節點本身) | O(1) (原地修改) |

---

#### 記憶體存取模式

##### 鏈表方法 (當前演算法)

```
記憶體存取序列：

初始掃描：
0xA000 ──▶ 0xA010 ──▶ 0xA020 ──▶ 0xA030 ──▶ 0xA040
[1]         [2]        [3]        [4]        [5]
順序掃描，不連續記憶體

修改操作：
step 1: 寫入 0xA000.next = 0xA020 (修改node1)
        寫入 0xA010.next = 0xA030 (修改node2)
        讀取 0xA020.next (讀取node3)
        讀取 0xA030.next (讀取node4)

記憶體躍跳：0xA000 → 0xA010 → 0xA020 → 0xA030 → 0xA040
                    (16bytes) (16bytes) (16bytes) (16bytes)
```

**特性：**
- ❌ **快取友好性低**：指標指向任意記憶體位置，造成快取不命中
- ✓ **單通過 (Single Pass)**：線性掃描效率O(n)
- ✓ **原地修改**：無額外陣列分配
- ❌ **指標追蹤開銷**：每次跳轉需解析指標

---

##### 陣列方法 (假設)

```
記憶體佈局（連續）：
╔═════════════════════════════════════════════════════╗
║ index 0  │ index 1  │ index 2  │ index 3  │ index 4 ║
║   val=1  │  val=2   │  val=3   │  val=4   │  val=5  ║
║ 0xC000   │ 0xC004   │ 0xC008   │ 0xC00C   │ 0xC010  ║
╚═════════════════════════════════════════════════════╝

記憶體存取序列：
陣列掃描：0xC000 ──▶ 0xC004 ──▶ 0xC008 ──▶ 0xC00C ──▶ 0xC010
                  (4bytes)   (4bytes)   (4bytes)   (4bytes)
```

**特性：**
- ✓ **快取友好性高**：連續記憶體配置，快取命中率高
- ✓ **單通過 (Single Pass)**：線性掃描O(n)
- ⚠️  **額外空間**：可能需要輔助陣列記錄奇/偶索引
- ✓ **記憶體存取簡單**：直接索引取得值

---

#### 快取行為分析

##### 鏈表方法快取行為

```
L1 快取行（典型 64 bytes）：

│ 0xA000 0xA004 0xA008 │ | 0xA010 0xA014 0xA018 │ | ... │
│   [node1 (16bytes)]  │ │   [node2 (16bytes)]  │ │ 空  │
└──── 命中 ────────────┘ └──── 命中 ────────────┘

但如果節點分散：
│ 0xA000 [node1] │ [空閒] │ 0xB000 [nodeX] │
└─ 命中 ─────────┘        └─ 冷啟動/重新讀取

快取不命中率：高 (取決於記憶體分配策略)
```

**預測：**
- 快取不命中每次跳轉
- 記憶體延遲：~100-300 CPU週期
- 總成本：指標追蹤開銷 × n

---

##### 陣列方法快取行為

```
L1 快取行（64 bytes）：

│ 0xC000 0xC004 0xC008 0xC00C 0xC010 0xC014 0xC018 0xC01C │
│    1      2      3      4      5    (剩餘空間)         │
└──────────── 完全命中 ──────────────────────────────────┘

順序掃描：完全在 L1 快取範圍內
```

**預測：**
- 快取命中率：~95% 以上
- 記憶體延遲：~4 CPU週期 (L1 命中)
- 總成本：最小

---

#### 原地鬼魂vs額外空間權衡

| 方面 | 鏈表 | 陣列 |
|------|------|------|
| **原地修改** | ✓ 直接改next | ⚠️  需臨時變數/標記 |
| **指標開銷** | ✓ 無額外陣列 | ❌ 可能需輔助陣列 |
| **修改成本** | ~8 次寫入 (example 1) | ~2n 次交換或移動 |
| **恢復難度** | ⭐⭐ (已分散) | ⭐⭐⭐ (可重新排序) |

---

#### 演算法複雜度對比

**鏈表方法 (實現)：**
```
時間複雜度：O(n)
  - 單次遍歷所有節點
  - 每節點恆定操作 (2次next賦值)

空間複雜度：O(1)
  - 只使用 3 個指標 (odd, even, starting_even)
  - 無遞迴堆疊或額外資料結構

快取複雜度：O(n) 記憶體跳躍
  - 最差快取行為：每節點造成 ~1 次快取不命中
  - 實際延遲：~100-150ns per node (現代 CPU)
```

**陣列方法 (假設實現)：**
```
時間複雜度：O(n)
  - 線性掃描 + 原地調整

空間複雜度：O(n) 最差
  - 可能需要 odd_arr, even_arr 等輔助結構

快取複雜度：O(1) 快取不命中
  - 完全順序存取
  - 實際延遲：~4-5ns per node (L1 命中)
```

---

#### 結論與應用場景

##### 何時使用鏈表方法（本題）

✓ 輸入已是鏈表形式（無轉換成本）
✓ 必須原地修改指標結構
✓ 記憶體配置器已最佳化（malloc pool）
❌ 對快取效能敏感的系統

##### 何時使用陣列方法

✓ 需要快取親和性（現代 CPU）
✓ 輸入是陣列或可轉換成陣列
✓ 記憶體頻寬限制環境（SIMD 優化空間）
❌ 無法預先知道陣列大小
